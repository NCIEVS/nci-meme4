inverse_relationships

Overview/Objective

This document contains a description of inverse_relationships and its various fields. It also (may) include information about how this table is used in the various stages of MID processing.
 

Details

Table name:inverse_relationships
Table Type: auxiliary,metadata
Description:Maps relationship names to their inverses. Additionally it allowed for the ranking of relationship names so that it can participate in the overall rank of the relationship. The weak_flag field can be used to indicate that some relationships (like RT?) are weaker relationships than others. This can be useful in terms of integrity checks like MGV_E which may be altered to say that a merge can be stopped if there exists a relationship between two concepts, unless that relationship is a weak relationship.
Fields:
relationship_name VARCHAR2(20) a valid relationship name, primary key 
inverse_name VARCHAR2(20) the inverse relationship name, e.g. the inverse of BT is NT 
weak_flag VARCHAR2(20) a Y/N flag indicating whether or not this is a weak type of relationship 
long_name VARCHAR2(100)  
release_name VARCHAR2(20)  
rank NUMBER(12) the rank of this row, higher values are better ranks 
Indexes: No indexes
Editing Role:When reading a concept from the database for a given concept_id, relationships must be read in both directions (where concept_id_1=concept_id and where concept_is_2=concept_id). When reading the inverse direction (where the current concept we want is the concept_id_2 of the relationship), we must reverse the sense of the relationship (e.g. broader becomes narrower) by mapping it through this table.
