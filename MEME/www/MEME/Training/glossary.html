UMLS Project Glossary

Following is a glossary of frequently used, potentially confusing terms.
 
 
action model
A scheme by which all changes made to core data are completely logged and tracked allowing "time travel" capability.  The MID stores only a single state for each data element, but tracks each action affecting those data elements as data itself.  This allows the database to be rolled back in time by undoing successive actions in reverse order, and then rolled forwards again by redoing those same actions. The advantage of an action model is that it minimizes storage overhead by not replicating states every time a change occurs.  The disadvantage of an action model is that time travel is computationally intensive. [See atomic action, molecular action, state model]

 
additional relationship label [aka relationship attribute]
Synonym of relationship attribute. 
 

AH bin
Ad-hoc bin created by the EMS.  These bins are not based on integrity checks, and are not created for the purpose of editing new/updated source content. Rather, they are things like the needsrelbin (which produces clusters of concepts which have ambiguous LUIs) which are generally useful but ad hoc in nature. [See ME bin, QA bin]

 
ambiguity
Typically refers to some unit of meaning appearing in multiple concepts. [See ambiguous string, ambiguous lui]

 
ambiguous string [aka. multiple meaning, separated string]
If the same string appears in multiple concepts, it is considered an ambiguous string.  This example (taken from Meta2001AA) shows the same string in three different concepts:
  C0010823|ENG|S|L0010825|VO|S0000811|Cytomegalovirus|3|
  C0010825|ENG|P|L0010825|PF|S0000811|Cytomegalovirus|0|
  C0677047|ENG|S|L0010825|VO|S0000811|Cytomegalovirus|3|
The test for ambiguous strings looks for case insensitive string matches across concepts.  The example above shows case sensitive matches but this does not have to be the case.  In the MID, ambiguous strings are found by looking up ISUI values that have different concept ids.  There are a number of important integrity constraints that apply to cases of ambiguous strings, in particular:
  • Every concept containing an ambiguous string should get a disambiguating MTH/PN atom.
  • For each ambiguous string case, every concept participating in that case must have a relationship to every other concept also participating.
  • MTH/PN atoms may not be ambiguous.  An MTH/PN atom in one concept containing an ambiguous string may have a string matching the ambiguous string, but no other concept in the MID may have a string matching that MTH/PN.
  • A concept may not have more than one MTH/PN atom.

 
ambiguous lui
If the same LUI appears in multiple concepts it is considered an ambiguous LUI.  Every case of an ambiguous string is implicitly a case of an ambiguous lui.  If a LUI is split across multiple concepts, it implies that there is some lexical similarity between the two classes, which means that the concepts are likely related in some way.  The NLM has an editing bin the goal of which is to have editors assign relationships between any two concepts that have an ambiguous LUI.  The following example (taken from Meta2001AA) shows a case of an ambiguous lui without an ambiguous string.
  C0010504|ENG|P|L0000003|PF|S0029254|Cyanidanol-3|0|
  C0917995|ENG|P|L0000003|PF|S0007489|(+)-Cyanidanol-3|0|
A review of the relationships shows that the second one is marked as narrower than the first.

 
approved concept [aka reviewed concept]
A concept that has been looked at by an editor and determined to be correct.  The concept becomes "approved" either by an explicit editor approval action or by an automated process that approves content that editors have OK'd on paper.  [See stamping]

 
associated expression
A boolean expression relating a concept to a combination of one or more other concepts.  Associated expressions are found in the MRATX file.  Consider this example (taken from Meta2001AA):
  C0000847|MSH2001|RO|<Pulse> AND <Foot>/<blood supply>|
This shows that the CUI in the first field is related to the combination of concepts in the fourth field (which are shown as strings instead of CUIs).  The third field is the type of relationship, in this case "related". 

 
atom [aka term]
An occurrence of a name in a source.  Also, considered the smallest unit of meaning in a concept.  Atoms are the noun phrases that make up the synonyms of a concept.  Atoms are also known as terms and sometimes as classes.  In the MID atoms exist in the classes, foreign_classes, atoms, and string_ui tables.  In the Metathesaurus, they exist in the MRCONSO, and MRX (index files).   Every atom is assigned a SUI, an ISUI (in the MID only), and a LUI. [See string]

 
atomic action 
A logged action that changes data in a single row of a core table.  Implemented by the MEME_APROCS package, these actions are logged in the atomic_actions table.  In addition to atomic actions that change core table fields, there is an atomic action to undo any other atomic action and an atomic action to redo any undone atomic action. [See molecular action]
 
 
AUI
A versionless atom unique identifier.  Any two atoms with the same SUI, root source, code, source_aui, source_cui, source_dui, and term type will have the same AUI value.  This values is assigned to new atoms in the MEME_SOURCE_PROCESSING.assing_auis procedure.
  
 
ATUI
A versionless attribute unique identifier.  Any two attributes with the attribute name, root source, attribute value connected to the same atom or concept will be assigned the same ATUI.  This values is assigned to new attributes in the MEME_SOURCE_PROCESSING.assing_atuis procedure.  [See Metathesaurus UI]
  
 
base string
A base string is a string belonging to an atom whose termgroup is something other than MTH/MM or MTH/TM.  More generally, it is a string that does not match an expression like (in perl):   / \<\d{1,}\>$/.  For example, 
  Osteopathic Manipulations <1>
is not a base string, but
  Osteopathic Manipulations
is a base string.  It can also be said that the second string above is the "base string" of the first string. In other words, a base string can also refer to a bracket string sans the bracket term part. [See bracket number, bracket string]


basic relationship label [aka relationship name]
Synonym of relationship name. 
  

batch action 
Implemented by the MEME_BATCH_ACTIONS package, batch actions are an efficient way of running a large number of similar molecular actions at once.  They log one molecular action and one atomic action for each data change made.  Batch actions are typically used by the source insertion process for doing things like making all of the atoms of an obsolete source unreleasable. [See macro action]

 
bequeathal relationship
A special type of relationship used to map a CUI that will be deleted to a CUI that will not be deleted.  Three special relationship names were created for this purpose: bequeathed broader (BBT), bequeathed narrower (BNT), and bequeathed related (BRT).  These relationships can be added by level 5 editors in the editor interface so long as the concept being bequeathed contains a single CUI value (i.e. the distinct count of non-null  last_release_cui values among the atoms in the  concept equals one).  In the MID, bequeathal relationships are stored in the relationships table.  In the MRD, bequeathal relationships are stored in relationships and mrd_relationships.  In the Metathesaurus, bequeathal relationships can be found in MRCUI.  In MEME4 (and beyond), bequeathal relationships will be directly managed in cui_map.

 
bin
A set of clustered concepts created by a particular predicate or set of predicates.  Bins are used as the basis for most worklists and checklists by the EMS and WMS. [See ME bin, QA bin, AH bin]

 
binary integrity check
Refers to an integrity check that requires two parameters.  For example, a check that enforces the rule "do not let <source a> merge with <source b> if their codes are different" requires two pieces of information, and so is a binary check. These checks make use of the ic_pair table in the MID. [See unary integrity check]

 
bracket number
In a string ending with an expression like (in perl):   / \<\d{1,}\>$/, it is the number in between the brackets.  For example, if we have the bracket string,
Osteopathic Manipulations <1>
the bracket number would be 1. [See base string, bracket string]

 
bracket term
The part at the end of a bracket string.  For example the ' <1>' in 'Apple <1>'.

 
bracket string
What a base string is not, a bracket string is.  A bracket string is a string matching an expression like (in perl):  / \<\d{1,}\>$/. [See base string, bracket number]

 
checklist
A table containing a list of (potentially clustered) concepts generated by the WMS.  Checklists are like worklists but are not not tracked as rigorously. They exist to facilitate ease of editing for things like QA bins and ad-hoc editing sets. [See worklist]

 
concept
The fundamental unit of the Metathesaurus.  A concept represents a single "meaning" and contains the various ways of expressing that meaning in the form of atoms.  All of the atoms within a concept are synonyms.  Each concept is assigned at least one semantic type, and possibly other attributes.  Every concept is assigned a CUI which (ideally) uniquely identifies a single "meaning" and is stable over time.

 
concept report
A content view is any definable subset of the Metathesaurus that is useful for a specific purpose.  Examples of content views are: the set of HIPAA vocabularies, a NLP (or NLU) subset.  For more details on content views click here.
  
content_view

A standard format for viewing the content in a concept.  Concept reports can be generated from by web here: /cgi-oracle-meowusers/concept-report-mid.pl.  Concept reports can be generated from the command line by running the $MEME_HOME/bin/reports.pl script.

context
Often atoms are given a context by the source providers that create them.  A context can be thought of as a hierarchy expressed as a path from that particular atom the root of its tree.  An context has a "self", an ancestor chain (leading to the root), child atoms, and sibling atoms.  During inversion, contexts are represented by .raw3 files and generally are built using the suite of tools in $MEME_HOME/Contexts.  In the MID, contexts are represented as parent  and sibling relationships in the context_relationships table. In the MRD, contexts are represented as a path to the root (as a chain of atom ids) in the mrd_contexts table. In the Metathesaurus, contexts are represented in the MRCXT file, and as parent, child, and sibling relationships in the MRREL file. [See context relationship]

 
context relationship
A parent, child, or sibling relationship between two atoms.  A parent means that one atom is higher in the context tree than the other atom.  A sibling means that two atoms have the same parents and paths to the root.  A child relationship is the inverse of a parent relationship, the other atom would be higher in the context tree than the one atom.  When inserting a source, .raw[23] context files are converted into context relationships by the $MEME_HOME/bin/raw_to_context.csh script. In the MID, they are represented in the context_relationships table.  The MRD has a context_relationships table which is used to generate data for the mrd_contexts (via the MRD_OPERATIONS.generate_auxiliary_data_states procedure).  In the Metathesaurus, context relationships appear as PAR, CHD, and SIB rows in MRREL. [See context]

 
co-occurence
A case where atoms of two different concepts appear in the same journal article, abstract, or other publication.  It is a co-occurence because two different concepts are occurring together in some third party source.  Typically we track the source of the co-occurence, an id representing the citations, frequency of co-occurence and type of co-occurence.  In the MID this data is represented in the two tables coc_subheadings and coc_headings.  In the MRD, this data is transformed from the coc_subheadings and coc_headings tables into the mrd_coc_subheadings and mrd_coc_headings tables (via the MRD_OPERATIONS.generate_auxiliary_data_states procedure).  In the Metathesaurus, co-occurences are represented (in various forms) in the MRCOC, MRSAT, and MRLO files.

 
core data
Data in the MID (or MRD) which contains actual content and can be directly edited by editors.  Core data includes atoms, concepts, attributes, and relationships.  context relationships occupy a pseudo core data status.  In future releases of MEME, core data will additionally include the translation atoms, co-occurence data, contexts, and CUI mappings. Core data has the following properties
  • Changed only by atomic actions
  • Tracked in code_map as a core table
  • Each row can be assigned a rank
  • Each row has editing fields: status, tobereleased, released, dead, rank, authority, timestamp, insertion_date, rank.
  • Live table has a corresponding dead table.

 
core table
A table that contains core data.

 
CUI
A Concept Unique Identifier.  One of the principles of the Metathesaurus is that meanings should be preserved over time regardless of what atoms are used to express those meanings.  The CUI is an identifier that uniquely represents a meaning and (ideally) over time the meaning of a CUI does not change.  As sources are updated and as editors merge and split concepts, meanings are altered and sometimes disappear.  A tracking scheme now exists (involving what are called bequeathal relationships) to map meanings that go away (deleted cuis) to living CUIs.  This data can be found in the Metathesaurus in the MRCUI file and in the MID and MRD in the cui_map table.  CUIs are algorithmically assigned on a daily basis by the MEME_OPERATIONS.assign_cuis procedure which uses relative rankings and previous cui assignments of atoms to determine which concept wins a CUI in a split and which CUI wins a concept in a merge.

 
data constraint
An integrity constraint designed to check some data condition.  Typically data constraints are checked after molecular actions complete to determine the post-action integrity state of the concept.  In rare exceptions (such as concept approval) violations of data constraints can actually stop actions from happening.  Data constraints all have names starting with DT_ (e.g. DT_M1).  [See merge inhibitor]

 
demoted fact [aka demotion]
A merge fact that has failed to produce a merge, and has thus been demoted.  Also a synonym for demotion.

 
demotion [aka demoted fact]
A type of relationship typically resulting from a failed attempt to merge two concepts.  Demotions are a flag for editors meaning that these two concepts are potentially synonymous but were not merged for integrity reasons.  Demotions are created during source insertions when performing automated merge sets.  A complex set of merge inhibitors is used to prevent merges from occurring under certain conditions (such as two concepts containing MSH main headings).  If an automated merge set is running and one of its merges fails due to these integrity constraints, a relationship is created between the two concepts that were to be merged; this is a demotion.   Infrequently, demotions are created without attempting to merge, especially in cases where we know that all of the attempted merges would fail.

 
editing interface
An application that allows editors to view and edit the content in the MID.  The molecular actions implementation exists in the back end of the interface and is the code that actually enacts the editor actions.

 
editor
A (presumably) qualified person who is a biomedical content expert and trained in using the various MEME editing tools.  Most of the editors are contractors who work for either Kevric or MSD.  Editors are assigned an editor level which grants them various levels of editing privilege.  The most privileged editors are called guru editors. 

 
embryo
A concept with a status of E.  These are concepts that are currently undergoing changes due to the source insertion process.  To avoid contaminating the process, editors are not allowed to edit concepts in an embryo state.  Once source insertion is complete, embryo concepts are converted into approved or unapproved concepts.

 
EMS
The Editing Management System is a web application designed by the NLM to manage the editing of concepts.  It enables the partitioning of the database into mutually exclusive sets which are then broken down into worklists.  The EMS is available at /cgi-oracle-meowusers/ems.pl  . [See WMS]

 
fact
Most generally, a fact is an assertion of some kind of relationship between two atoms.  A fact  can be used to merge those atoms, to create relationships between those atoms, to assert that one is a safe replacement of the other, or even something else.  [See generated fact, precomputed fact]

 


full update insertion
An  update source insertion where an entire copy of the new source is inserted.  This will result in more than one version of a source being present in the MID at a given time, in which case one is considered the current version  [See partial update insertion]
 
 
generated fact
A fact generated through a process called matching.  Typically generated facts are employed during the source insertion process to algorithmically merge atoms, or to create relationships (usually demotions) between atoms. [See precomputed facts]

 
hypernym
A "near synonym" of an atom that has a broader meaning. [See hyponym]

 
hyponym
A "near synonym" of an atom that has a narrower meaning. [See hypernym]

 
ic code
This refers to the code assigned to an integrity check within an integrity vector that describes the level of enforcement of that check within that vector.  Its legal values are given in the code_map table where the type is integrity_vector. [See ic state]

 
ic state
This refers to a state assigned to an integrity check within an integrity vector once it has been determined that the check produced a violation.  The code_map table defines the transformation from ic code values to ic state values in the rows where type is ic_state.

 
ISUI
A case insensitive string  unique identifier.  Any two strings (within the same language) that have the same case insensitive string will have the same ISUI value.  This values is assigned to new atoms in the MEME_SOURCE_PROCESSING.assign_string_uis procedure. [See LUI, SUI]

 
integrity check [aka integrity constraint]
This is a synonym for integrity constraint.

 
integrity constraint [aka integrity check]
The MEME integrity system is built on a series of integrity constraints that are designed to look at concepts and determine if certain conditions exist, or to examine certain pairs of concepts and determine if certain conditions exist.  Integrity constraints are used to prevent merges by the merge engine, they are used by the matrix initializer to unapprove concepts that violate particularly severe conditions, and they are used by the EMS to generate checklists for editor review.  Each constraint has a name and is mapped to a particular procedure (in code_map).  For complete documentation on the active integrity checks see these two links:  integrity system, MEME_INTEGRITY_PROC. In the MID, the integrity constraints are managed in these tables: integrity_constraints, ic_applications, ic_override, ic_single, and ic_pair.  [See data constraint, merge inhibitor, unary integrity check, binary integrity check]

 
integrity vector
Generically, an integrity vector is a collection of integrity checks each assigned an ic code and used by an application to determine which constraints to check and at what enforcement level.  Vectors have a syntax that involves concatenating a check name, a colon :, and an ic code and then wrapping that in < and >.  Any number of these can then be appended together.  For example:
<MGV_A1:E><MGV_A2:E><MGV_A4:E><MGV_B:W><MGV_C:E>
This shows a vector with five checks.  Three of them have an ic code of E, indicating enforcement, and the other has a code of W indicating that violations should produce only a warning.   Integrity vectors are stored in the ic_applications and ic_override tables. The semantics of these two tables allows checks listed in a vector from ic_override to have their enforcement level override that of the same check in ic_applications. [See override vector, violations vector]

 
inversion
The complex process by which files sent from source providers are converted to our common format. [See src files]

 
lexical class
A classification of strings based on lexical similarity.  A lexical normalization program (such as lvg) will compute a transitive closure over the strings such that each string will have a normalized string, and strings with common normalized strings are said to have the same lexical class.

 
lexical variant
A string that is lexically similar to another string.  Common lexical variants are plural vs. singular, capitalization differences, and word order differences.  For example,

 
Abbreviated injury scale  and  Abbreviated Injury Scales

 
are pluralization and case variants of one another.  Here we have a word order variant,

 
Abbreviated Injury Scale  and  Injury Scale, Abbreviated

 
Lexical variants of this nature are determined by passing various flows to the NLM's lvg program. [See string type]

 
LUI
A lexical class unique identifier.  Any two strings (in english) that have the same normalized string after having been run through the NLM's luiNorm program will have the same LUI.  For foreign language strings, the normalized string is identical to the string, so no two SUIs will have the same LUI within the same language (although they may have the same ISUI). LUIs are assigned by the MEME_SOURCE_PROCESSING.assign_string_uis procedure. [ See ISUI, lexical class,  SUI]

 
luiNorm
A version of the lvg program norm which produces a single normalized strings for any given input string.  This program is used to generate normalized strings for LUI assignment. [See norm]

 
lvg
A suite of lexical normalization tools part of which is the specialist lexicon a general english language lexicon that includes many biomedical terms. [See luiNorm, norm]

 
macro action 
Implemented by the MEME_BATCH_ACTIONS package, macro actions are used to perform a large number of atomic actions in a way that can be efficiently reported to the MRD.  They log only a single row in molecular_actions for many atomic actions.  Macro actions are typically used by the source insertion process for doing things like inserting all of the atoms of an new or update source. [See batch action]

 
main heading
The term type associated with the preferred terms in the D# portion of MeSH. [See preferred term]

 
matching
A process for generating pairs of atoms that are related in some way.  There are three main functions of matching:
  • merge concepts in separate atoms
  • create relationships between atoms
  • calculate safe replacements between atoms of an obsolete source and an update source.

 
The process of matching involves joining the classes table on itself (or occasionally on source_classes_atoms).  There are four typical matching predicates, or criteria.  They are:
  • string
  • code
  • term type
  • concept id
  • source concept id (source_cui)

 
The following table shows the different possibilities for these four predicates and has descriptions of what they mean.

 
String EXACT Look for cases where case insensitive strings are the same (isui=isui).
NORM Look for cases where normalized strings are the same but case insensitive strings are not (lui=lui AND isui!=isui).  This should probably be called NORM_ONLY.
BOTH Look for cases where normalized strings are the same. (lui=lui).
NOT Look for cases with different case insensitive strings (isui!=isui).
NONE Do not restrict further (1=1).
Code EXACT Look for cases where codes match (code=code).
NOT Look for cases where codes differ (code!=code).
NONE Do not restrict further (1=1).
Term Type EXACT Look for cases where term types are the same (SUBSTR(termgroup,INSTR(termgroup,'/')+1)= SUBSTR(termgroup,INSTR(termgroup,'/')+1))
NONE Do not restrict further (1=1)
Concept id EXACT Look for cases where concept ids are the same (concept_id=concept_id)
NOT Look for cases where concept ids are different (concept_id!=concept_id)
Source CUI
 EXACT
 Look for cases where source cuis match (source_cui=source_cui)
NOT
 Look for cases where source cuis differ (source_cui!=source_cui)
NONE
 Do not restrict further

 
The point of breaking it down this way is that knowing what the table above means, we can write code to dynamically generate the necessary sql query to create the matches we are interested in. One such example of this dynamic code is MEME_SOURCE_PROCESSING.generate_facts which is responsible for generating merge facts based on a set of parameters.

 
matrix initializer
A batch process which reviews every concept in the database and determines the editing status of the concept.  It looks at the various parts of the concept (atoms, relationships, attributes), and then performs a series of integrity checks looking for serious violations.  If there is any reason to believe that the concept requires further editing, the concept is unapproved.  Additionally, the matrix initializer renormalizes the releasability of the concept, based on the releasability of its constituent atoms.  The matrix initializer is run with the $MEME_HOME/bin/matrixinit.pl script, but requires the MEME_INTEGRITY.matrix_initializer procedure.  There is another form of this code called the matrix updater which is more efficient because it "catches up" from the last run rather than looking at the entire database.

 
ME bin [aka mutually exclusive bin]
Mutually exclusive bin created by the EMS when the MID is repartitioned.  There are six permanent mutually exclusive bins (demotions, embryos, norelease, noreview, reviewed, leftovers)  and other ones are added, generally to create bins containing concepts from a new or update source.  [See AH bin, QA bin]

 
MEME
Metathesaurus Enhancement and Maintenance Environment.  MEME is a suite of tools written in a combination of Oracle PL//SQL, CA-OpenROAD, and Java that comprises the editing environment for the UMLS Metathesaurus.  Its tools include (but are not limited to)a back end implementation of the molecular actions, an editing interface, source insertion tools, and code for generating concept reports.  Each server that has the MEME system installed should have a $MEME_HOME directory which is the base directory for all of the code.  For a further breakdown of the MEME environment, see: /MEME/Training/it_environment.html.  MEME is currently on its third implementation with MEME4 on the way.  The home directory for the most current version of the MEME documentation is :  /MEME .

 
meme id
Every row of each core data table is assigned an id that uniquely identifies that row.  That id is a meme id.  In the classes and foreign_classes tables, they are called atom_id, in the attributes table they are called attribute_id, in the relationships and context_relationships tables they are called relationship_id, in the concept_status table they are called concept_id.  Actions that affect core tables are logged in the atomic_actions table which has a field called row_id which always contains a meme id.

 
merge
Occurs when a concept called a "source concept" is entirely moved into a "target concept".  It is an action that makes two concepts into a single concept.  At the implementation level, it changes all instances of the source concept's concept_id (in core tables) into the target concept's concept_id.  Merges can be performed by editors (in the editing interface) or by the merge engine (during source insertion). [See merge inhibitor, split, split-merge]

 
merge engine
Software that takes merge facts from the mom_merge_facts table and attempts to merge the corresponding atom_ids into the same concept.  There are four possible outcomes of this attempted merge:
  • The merge succeeds.  In this case the status field of the table is set to M.
  • The concepts are already merged.  In this case, the status field is set to P.
  • The merge fails due to integrity violations.  The merge engine is run with an integrity vector, and if any of the checks in that vector have an ic code of E and produce a violation, the merge is stopped.  In this case, the status field is set to D, for demotion.
  • The engine determines that the merge will fail.  If the merge engine knows that a merge will fail, it does not attempt it.  In this case, the status field is set to F.

 
There is a process leading up to actually attempting the merge that involves first loading facts into the mom_candidate_facts table.  This is done either by generating or loading the facts from a file.  Once facts are in the mom_candidate_facts table, they are loaded into mom_merge_facts.  Once the merge engine is finished, it (optionally) creates demotions for all facts marked with a D, and then moves the facts to mom_facts_processed. [See merge fact, generated fact, precomputed fact]

 
merge fact
A fact asserting that two atoms should be merged (if they are in different concepts).  Merge facts are either generated or loaded during source insertion before invoking the merge engine. [ See generated fact, precomputed fact]

 
merge inhibitor
A type of integrity check which looks at two unmerged concepts and determines if the merging would create an illegal condition.  Merge inhibitors are typically used in the integrity vectors passed to the merge engine to prevent illegal merges from happening.  Typically, when a merge inhibitor stops a merge, it results in a demotion.  One example of a merge inhibitor is a check that disallows MeSH main headings from merging.  For a complete list of merge inhibitors, review the integrity system documentation. Merge inhibitors all have names starting with MGV_ (e.g. MGV_A4) and in many cases are unary or binary integrity checks. [See data constraint]

 
merge set
During source insertion, merging operations are divided into logical sets called merge sets.  A merge set is a collection of merge facts that has a similar purpose, such as merging things known to be synonymous, or merging abbreviations with long forms.  Merge sets are given names which are tracked in the various mom_* tables, this allows someone to look up and see what particular merge set caused a pair of concepts to merge.

 
MetamorphoSys
A java client tool used for subsetting the Metathesaurus.  Users of the Metathesaurus may not have licenses for all available sources and so may be forced to exclude them from their distribution.  Alternatively, users may want to exclude sources for other reasons, or they may want to change the ranking of the various sources, or to alter the default settings for what source, term type combinations are suppressible.  MetamorphoSys takes a Metathesaurus distribution and allows users to make these kinds of selections and then it produces a subset with the user-defined rankings and suppressibility choices.  MetamorphoSys also supports a variety of input and output formats designed to allow users to access their subsetted data however they can best use it.  There is an open API for extending the range of formats and filters used by MetamorphoSys.

  
Metathesaurus
From www.nlm.nih.gov/pubs/factsheets/umlsmeta.html: "The UMLS Metathesaurus is one of three knowledge sources developed and distributed by the National Library of Medicine as part of the Unified Medical Language System (UMLS) project. The  Metathesaurus contains information about biomedical concepts and terms from many controlled  vocabularies and classifications used in patient records, administrative health data, bibliographic and full text databases and expert systems. It preserves the names, meanings, hierarchical contexts, attributes, and inter-term relationships present in its source vocabularies; adds certain basic information to each  concept; and establishes new relationships between terms from different source vocabularies." [See UMLS, Original Release Format, Rich Release Format]
 
 
Metathesaurus UI [aka "metathesaurus identifier"]
Each data element that appears in the Metathesaurus Rich Release Format is assigned a unique identifier designed to remain stable over time, and especially through the updates of the various sources.  These identifiers serve as row_ids within the files as well as unique identifiers that can be tracked over time, even when a source is updated. [See AUI, ATUI, RUI]
 
 
MID
Metathesaurus Information Database.  This is the editing database which contains all of the data which will ultimately be released.

 
molecular action 
A logged action corresponding to a "high level" editor action that affects a whole concept and not just a single core table row.  Molecular actions employ one or more atomic actions to enact a higher level change, like delete a concept or merge concept a into concept b.  These actions are logged in the molecular_actions table and like atomic actions can be undone and redone. [See batch action, macro action]

 
mid service
The NLM maintains a socket server accessed by /site/umls/scripts/midsvcs.pl that maps a list of logical names to values.  A mid service can generically refer to any value associated with a key (running the script without any arguments will generate the list of valid keys).  More specifically (for MEME4), a mid service refers to any of the generic mid service keys that end in -jdbc or -tns that can be used to connect to a MID database.  In this more specific context, the mid service is the key without the suffix, for example current-editing-tns becomes the MEME4 mid service current-editing.

 
MRD
Metathesaurus Release Database. This database is a continuous and historical release environment for the data in the MID.  As data in the MID is changed, it is synchronized with the MRD, which stores this data in an intermediate state that is more like the final release product (the MR files).

 
MTH/MM
A special termgroup for bracket string atoms intended to disambiguate an ambiguous string.  If an ambiguous string exists in a concept that does not have a matching MTH/MM atom (meaning that the lowercase strings are the same, sans the bracket term), then an MTH/MM term is added to disambiguate the ambiguous string.  Consider these rows from Meta2001 MRCON and MRSO:
MRCON:
C0000165|ENG|P|L0000165|VO|S1465240|17-hydroxysteroid dehydrogenase <1>|0|
C0000165|ENG|P|L0000165|VS|S0783695|17-hydroxysteroid dehydrogenase|0|
C0525733|ENG|P|L0000165|PF|S0783695|17-hydroxysteroid dehydrogenase|0|
C0525733|ENG|P|L0000165|VO|S1465241|17-hydroxysteroid dehydrogenase <2>|0|

MRSO:
C0000165|L0000165|S0783695|AOD99|DE|0000018471|0|
C0000165|L0000165|S1465240|MTH|MM|U005031|0|
C0525733|L0000165|S0783695|MSH2001|CE|C035507|0|
C0525733|L0000165|S1465241|MTH|MM|U005032|0|
In this case, "17-hydroxysteroid dehydrogenase" is an ambiguous string so the "17-hydroxysteroid dehydrogenase <1>" and "17-hydroxysteroid dehydrogenase <2>" atoms were added to disambiguate the ambiguous string. Looking at the MRSO rows, you can see that they have termgroups of MTH/MM.

 
 There is a related termgroup, MTH/TM which is assigned to atoms that are place holders for MTH/MM atoms while editing is still underway.  The most complicated thing about MTH/MM atoms is their ranking scheme.  MTH/MM atoms do not have their own rank, rather their rank is "just above" the rank of the highest ranking atoms in the same concept that has a matching string (sans the bracket term).  If there are multiple MTH/MM atoms with the same base string, the one with the lower bracket number outranks the other one.  If a concept contains an MTH/MM atom whose base string matches no other strings in the concept, then it gets a rank of zero.

 
multiple meaning [aka. ambiguous string, separated string]
Synonym of ambiguous string.  Also, MTH/MM atoms are often called "multiple meaning atoms".

 
mutually exclusive bin [aka ME bin]
Synonym of ME bin.

 
norm
A program that is part of the lvg distribution which generates one or more normalized strings given an input string.  The program is implemented as an lvg call with a specific pattern of parameters. [See luiNorm]

 

new source insertion
A source insertion involving content that has never appeared in the Metathesaurus.  [See update insertion]
 
 
normalized source
A field of the source_rank table used for conversion of some source values during production.  In certain cases, source inversion produces source values that are not valid source for a release and must be converted to valid sources.  For example, some relationships connected to the C code part of MSH have MSH2001HMCE as their source.  The actual valid source name is MSH2001, so there would be an entry in source_rank where the source field would be MSH2001HMCE and the normalized source would be MSH2001.  During production, any sources that are encountered in the core tables are converted to the normalized source if the source does not equal the normalized source.

 
normalized string
A string is converted to a normalized string by running it through norm or luiNorm.  Normalized strings appear in the MID in the string_ui and normstr tables.  In the MRD, normalized strings appear in string_ui, normstr, and mrd_normstr which is an MRD transformation of the normstr table.  In the Metathesaurus, they appear in the MRXNS.ENG file. Note: normalized strings are also broken into normalized words and put into the normwrd table in the MID.  In the MRD, normalized words appear in normwrd and mrd_normwrd.  In the Metathesaurus, they appear in the MRXNW.ENG file.

 
obsolete source
A version of a source that already exists in the MID that is being replaced by an updated version. [See update source]

  

original release format [aka "MR Files"]
The Metathesaurus is released as a series of relational flat-files in a specific format.  This refers to the original format of those files which included such vagueries as splitting atom information into two files (MRCON, MRSO) and obscuring atom level identifiers from the attributes and relationship files.  The format was used for over 10 years with minor modifications.  This format is still supported by MetamorphoSys.   [See Rich Release Format, Metathesaurus]
 
 
override vector
A type of integrity vector taken from the ic_override table indented as a mask to override a vector taken from the ic_applications table. Suppose an application takes a vector from ic_applications and it contains the check <DT_M1:E>.  If the application then uses a vector taken from ic_override that contains the check <DT_M1:W>, the enforcement level of the check is changed from E to W.  Typically override vectors are used to give gurueditors special privileges.   [See violations vector]
 
 


partial update insertion
An  update source insertion where only the set of things that has changed since the previous version is inserted.  This type of insertion requires either the source itself or the inversion process to have an update model for the source.  After the insertion, only those things from the old version that changed or were removed from the current version will still be in the MID, all other data elements will appear as part of the current version. [See full update insertion]
 
 
partitioning 
A process performed by the EMS that divides all of the concepts in the MID into mutually exclusive sets.  There exists a list of predicates which are run in order, each of which syphons off concepts into a bin which are not allowed to exist in the later sets.  The result of partitioning is the ME bins.  Each time a new or update source is inserted,   predicates for extracting concepts belonging to that source are added and the MID is repartitioned.

 
polysemy [aka ambiguous string, multiple meaning, separated string]
A case of  ambiguous strings.

 
pre-insert merge
During source insertion, updated versions of a source are merged with obsolete versions of the source to take advantage of the past editing of the old source.  As an optimization, we sometimes do this mergingbefore loading the updated source into the core tables.  This is implemented as an update of concept_ids on the source_classes_atoms table.  Although much faster, using this technique carries risks because integrities cannot be enforced to prevent merges

 
precomputed fact
A merge fact loaded from a file.  Merge facts can either be generated or computed during inversion and put into a file called mergefacts.src.  This file is loaded into the mom_precomputed_facts, processed by MEME_SOURCE_PROCESSING.prepare_src_mergefacts, and then loaded into mom_candidate_facts where they are processed by the merge engine. [See generated fact]

 
preferred name [aka preferred term]
The string associated with the highest ranking atom for some grouping of atoms, typically a concept. Note: any grouping of atoms can have a preferred name, for example the highest ranking atom in a MeSH D# is the preferred name of that D#.

 
preferred term [aka preferred name]
Synonym of preferred name.

 
production
The complex process for creating a release.  Eventually, production will be performed as an MRD process. [See release]

 
QA bin
Quality assurance bin created by the EMS.  These bins correspond to various integrity checks implemented as MEME_INTEGRITY_PROC procedures. [See AH bin, ME bin]

 
rank
A value associated with some piece of information that imposes an ordering scheme on like pieces of information.  Almost everything in the system is ranked, including fields like released, tobereleased, status  etc.  The most important instance of rank is with respect to the core data.  Atoms are ranked to determine the preferred name for a concept (or the preferred term for a LUI within a concept).  Relationships are ranked to determine the preferred relationship between a pair of concepts.   Rankings of core data are based on the relative rankings of their constituent parts.  All ranking within the system is handled by the MEME_RANKS package.


recipe
A document describing how to take inversion files and load them into a MID in such a way as to minimize editing.  The recipe takes three forms, an HTML document viewable here, an XML document that represents the java object model and can be reloaded into an application for editing recipes, and a .csh script that can be run to actually perform an insertion.  Eventually the shell script will be phased out in favor of a purely web-based solution. [See source insertion]
 
 
relationship attribute [aka additional relationship label]
A further specification of a more generic kind of relationship.  For example, a narrower relationship can be further specified as isa or has ingredient. Valid relationship_attribute values in the MID relationships table exist in the inverse_rel_attributes table.  Generally, relationship attribute values can be found in the Semantic Network.  In the Metathesaurus, these values are found in the RELA field of the MRREL file. 
 
 
relationship label
Generically refers to the nature of a relationship.  It is encoded in the Metathesaurus MRREL file in the REL and RELA fields. In the MID, it is encoded in the relationship_name and relationship_attribute fields of the relationship table. [See basic relationship label, additional relationship label]

   
relationship name [aka basic relationship label]
The high-level nature of a relationship.  For example, a relationship can express that one thing is narrower then another. Valid relationship_name values in the MID relationships table exist in the inverse_relationships table.   In the Metathesaurus, these values are found in the REL field of the MRREL file. 
 
 
releasable
Core data is said to be releasable if it will appear in the next version of the Metathesaurus.  Releasability is indicated by a tobereleased value in the respective core table of Y or y.  [See unreleasable]

 
release
An instance of the Metathesaurus.  Releases are assigned version numbers like this 2001AA, or like this 2001AA_01.  The first form indicates an unsubsetted release, and AA indicates the first version of the year 2001.  2001AB would indicate the second year.  2001AA_01 indicates a subset of the first version of the 2001 release. [See production]

 
restriction level
Every source has a restriction level between 0 and 3.  The level indicates the type of license agreement that a user must have to use that source.  0 is the least restrictive (meaning the source is free or in the public domain) and 3 is the most restrictive (meaning that you have to pay a license fee to the source provider).  Typically, users remove source they do not have licenses for by using MetamorphoSys.

 

reviewed concept [aka approved concept]
Synonym of approved concept.
  
 
rich release format [aka "MR Plus Files"]
The Metathesaurus is released as a series of relational flat-files in a specific format.  This refers to the rich format of those files implemented to extend the data model to support a finer granularity and an larger set of data types.  In part the motivation was to fully support SNOMEDCT, and in part it was a natural evolution of the data itself (initially envisioned by WTH as the "big pipe").  The format eliminates many problematic aspects of the original style files by such things as including Metathesaurus UIs in the data files, using a single file to represent the atoms (MRCONSO), increasing the granularity of files like MRREL to the atom level, and the addition of a field to support "Content Views" of the data. This format is supported by MetamorphoSys as of 2004AA.   [See Original Release Format, Metathesaurus]
 
 
root source
Typically, the root source is a source abbreviation stripped of its version information.  For example, the root source for MSH2001 is MSH.  The purpose of the stripped source is to have a source value that can be preserved across versions of the Metathesaurus.  MSH will identify MSH2001 in one version and MSH2002 in a later version, for example.  A root source should have the property that it should uniquely represent a single current SUI, source, term type, and code in the MID.
  
 

RUI 
A versionless relationship unique identifier.  Any two relationship with the relationship name, root source, and relationship attribute value connected to the same atoms or concepts will be assigned the same RUI.  This values is assigned to new relationships in the MEME_SOURCE_PROCESSING.assing_ruis procedure.  [See Metathesaurus UI]
 
 
SAB [aka source abbreviation]
Acronym for source abbreviation.
 
  
safe replacement
When a source is updated, there are atoms from the new source that are new versions of atoms from the old source, these atoms are safe replacements.  In practice, we calculate safe replacements by generating safe replacement facts (through matching), and then we store the safe replacement facts in a table called mom_safe_replacement.  It is possible that the matching criteria turns up multiple replacements in the new source for an old source atom, so we rank the safe replacement facts so that other applications can find the "best one".  Safe replacement atoms inherit last_release_cui and last_release_rank values from the atoms they replace.  This calculation is performed by MEME_SOURCE_PROCESSING.safe_replacement.

 
Semantic Network
A part of the UMLS which contains a hierarchy of the valid semantic types and relationship attributes.

 
semantic type
A piece of information that gives a concept a general category of meaning.  Semantic types are hierarchically arranged in the Semantic Network and there is a requirement that every concept have at least one semantic type.  In the MID, semantic types can be found in the attributes table.  In the MRD, semantic types can be found in attributes and mrd_attributes.  In the Metathesaurus, semantic types are found in MRSTY.

 
separated string [aka ambiguous string, multiple meaning]
Synonym of ambiguous string.  It is called a separated string because the same string appears in multiple concepts.

 
source asserted UI [aka "source asserted identifier"]
Sources may assert their own identifiers attached to the various data structures supported in the MID. [See Source AUI, Source ATUI, Source CUI, Source DUI, Source RUI]

 
source abbreviation [aka SAB]
Each source added to the MID is assigned an abbreviation that represents the source provider.  For example, MSH2001 is the source abbreviation for the 2001 version of the Medical Subject Headings.  All of the source fields in appearing in tables in the MID contain SAB values.
 
  
source AUI [aka SAUI]
A source asserted atom identifier.  Sources which assert their own atom identifiers are responsible for maintaining them over time and they are passed through to the Metathesaurus (rich release format) as fields in the MRCONSO file. [See Source Asserted UI]
  



source ATUI [aka SATUI]
A  source asserted attribute identifier.  Sources which assert their own attribute identifiers are responsible for maintaining them over time and they are passed through to the Metathesaurus (rich release format) as fields in the MRSAT file.. [See Source Asserted UI]
  
 


source CUI [aka SCUI]
A  source asserted concept identifier.  Sources which assert their own concept identifiers are responsible for maintaining them over time and they are passed through to the Metathesaurus (rich release format) as fields in the MRCONSO file.. [See Source Asserted UI]
  
 


source DUI [aka SDUI]
A  source asserted descriptor identifier.  Sources which assert their own descriptor identifiers are responsible for maintaining them over time and they are passed through to the Metathesaurus (rich release format) as fields in the MRCONSO file.. [See Source Asserted UI]
  
 

source RUI [aka SRUI]
A  source asserted relationship identifier.  Sources which assert their own relationship identifiers are responsible for maintaining them over time and they are passed through to the Metathesaurus (rich release format) as fields in the MRREL file.. [See Source Asserted UI]
 
 
source family
In some cases, different sources belong to the same family of sources.  What this means is that one source should not exist without the other one, so when a user is subsetting the Metathesaurus using MetamorphoSys, a decision to remove one of the sources in a family should also prompt a decision to remove all others in that family.  A classic example of different sources in the same family are CPT and MTHCH (the MTH created hierarchical terms for CPT).
 
 
source group identifier [aka generic identifier]
This kind of identifier was originally envisioned to represent natural groupings within a source (such as grouping by D#'s in MSH).  Its purpose has expanded to generically include any kind of identifier.  The complete list can be found by looking up rows in code_map where the type is sg_type.  Starting in MEME3, relationships and attributes no longer have to be connected to atoms or concepts, they can be connected to any valid type of source group id.  Internally, these source group ids are mapped down to the preferred term that represents the group and that atom and its concept are used in the core tables.  Mapping of source group ids to atoms and concepts is performed by MEME_SOURCE_PROCESSING.map_sg_fields. Eventually, we will rename these "generic identifiers".

 
source id
The inversion process creates src files as its end product.  Each row of the core data src files contains a unique identifier called a source id.  Each file is loaded into the database and each row is then assigned a meme id.  Mappings between source ids and meme id exist in the source_id_map table.  This table allows a user to take a particular core table row and look up the exact line in the src file that the core data came from.  [See meme id]

 
source insertion
The complex process of inserting new or update data into the database.  The process typically starts with the writing of an insertion recipe which describes how the src files will be loaded into the MID and merged with the existing content. [See inversion, new source insertion, update source insertion]

 

source of label
The relationships and context_relationships tables have two source fields.  These two fields correspond to the SAB and SL fields of MRREL.  The first source field indicates the source that asserted that a relationship exists between the two atoms/concepts.  The source of label indicates the source that asserted the nature of the relationship (broader vs narrower, etc.). The source of label always has a valid source value.
 
 
source transparency
The principle of  passing a source through the MID and the Metathesaurus without any loss of information.  There are three key aspects to source transparency:  (a) no information loss,  (b) documentation detailing the transformation from input format to Metathesaurusrrepresentation, and (c) semantics of the original source are preserved as much as possible.  The Metathesaurus is designed as a homogenous representation of vocabularies from a variety of different source vendors.  Source transparency guarantees that the transformation of a source from its native format to this homogeneous format does not truncate or obfuscate the data.
 
 
split
Whereas a merge puts two concepts together, a split tears a concept apart into two concepts.  A split happens when an editor decides that one or more atoms in a concept do not mean the same thing as the other atoms in the concept and removes those atoms from the concept and puts them in a new concept. Note: split is a term that generally refers to concepts, but generically it is a case of some constituent part of something larger being moved out.  When comparing one version of the Metathesaurus to another, you will often see CUIs that have been split apart. [See split-merge]

 
split-merge
This is a case of a split and a merge overlapping each other.  Suppose you have two concepts (C1 and C2) each of which has two atoms (A11, A12 in C1 and A21, A22 in C2).  Then suppose that A12 is split out of C1 (creating concept C3) and A22 is split out of C2 (creating concept C4).  Finally, suppose that concepts C3 and C4 were merged together.  We would have a situation where two concepts had parts of them split out, but those split out parts then merged with something else.  That is a split-merge. [See split, merge]

 
src files
The inversion process takes source provider files in their native formats and converts them to our common format which are the src files.  These files are then the input to the source insertion process.  Full documentation for the src files is available at:  /MEME/Data/src_format.html . 

 
stamping
An automated process for approving concepts. Editors are assigned worklists which are used to generate paper concept reports.  Editors review the paper reports and use the editing interface to make required changes.  An editor implicitly approves a concept on a worklists if they make no explicit changes to it in the interface.  Those implicit approvals are made explicit by the stamping process.

 
state model
A scheme by which all states of core data are preserved over time.  In a state model, actions are not logged as data, but rather cause the addition of a new state of the data changed, which typically will expire the old state.  The MRD employs a state model for tracking the release state of every core data element over time.  The advantage of a state model is that all states of the data are available at all times without computation.  The disadvantage of a state model is that it requires an increasingly large amount of space. [See action model]

 
string
A collection of characters that name something.  An atom represents a named thing but has lots of extra information associated with it.  A string is just the name part of an atom.

 
string type (STT)
A field of the Metathesaurus file MRCONSO, the string type indicates one of two things: (a) that a string is the preferred name for its LUI within its concept, or (b) what type of lexical variant a string is of the preferred name for its LUI within its concept.  In the first case, that row of MRCONSO would get an STT value of PF.  In the second case, that row of MRCONSO would get an STT starting with V and having one or more of the following qualifiers C (representing a case variant), W (representing a word order variant), P (representing a pluralization variant), or O (representing some other kind of variant). [See term status]

 
stripped source
Synonym of root source.

 
SUI
A case sensitive string unique identifier.  Any two strings (within the same language) that have the same case sensitive string will have the same SUI value.  This values is assigned to new atoms in the MEME_SOURCE_PROCESSING.assing_string_uis procedure. [See LUI, ISUI]

 
supplementary chemical [aka supplementary concept]
Synonym of supplementary concept.  This is what we used to call them.

 
supplementary concept [aka supplementary chemical]
A concept containing only MSH atoms with C# codes with a status of U.  Eventually the MSH people will edit all of the status U content and there will no longer be supplementary concepts.

 
suppressible
A flag attached to core data indicating that it exists for completeness but is ugly enough as to not be very useful.  The 40 character RCD abbreviations are a good example of this.  At present, there is only one dimension of suppressibility: either something should be suppressed, or it should not be, but eventually the notion of suppressibility will be expanded to provide the richer filtering capabilities.  In the MID and MRD, the core tables all have suppressible fields.  In the Original Release Format, the only way in which suppressibility is used is in the TS field of MRCON, where a value of s indicates that the row can be safely ignored.  In the Rich Release Format, most data files have a SUPPRESS field that carries this information.

 
synonym
Atoms that have the same meaning are synonyms.  In theory, concept-ness is the transitive closure of synonymy over all of the atoms, meaning that any two atoms appearing in the same concept mean the same thing, and therefore are synonymous. 

 
term [aka atom]
Generally synonymous with atom.  But in a stricter, more technical sense, a term is a CUI,LUI tuple in MRCONSO and so is broader than an atom is.

 
term status (TS)
A field of the Metathesaurus file MRCONSO, the term status indicates one of three conditions: (a) a LUI is the preferred LUI of its CUI, (b) a LUI is not the preferred LUI of its CUI and is not suppressible, and (c) a LUI is not the preferred LUI of its CUI and it is suppressible.  The first condition is given a value of P, the second a value of S, and the third a value of s. If a row in MRCONSO has a TS value of P and an STT value of PF, it is the preferred term for that concept, language tuple.  [See string type]

 
term type
A value indicating the general nature of an atom.  Term types include things like PT for "preferred term" or SY for "synonym" or MH for "main heading".  Term types can be found at the end of termgroup values after the slash. [See termgroup]

 
termgroup
A piece of information indicating the source and term type of an atom.  Termgroups have the microsyntax: <source>/<termtype>.  For example, MSH2001/MH indicates that an atom has the source MSH2001 and the term type MH.  The termgroup is redundant but exists for convenience.

 
transitive closure
An extension or superset of a binary relation such that whenever (a,b) and (b,c) are in the extension, (a,c) is also in the extension.  What?  This means is that if there is some binary relationship (like parent-child) that exists between a and b and also exists between b and c, then the transitive closure of that binary relationship over the set asserts an extended form of the binary relationship between a and c (if the binary relationship is parent-child the extended transitive relationship is descendent).  Another example is this, if a and b have the same lexical class and b and c have the same lexical class, then the transitive closure of lexical similarity over the set asserts that a and c must have the same lexical class.

 
UMLS
"The Unified Medical Language System project project develops Knowledge Sources that can be used by a wide variety of applications programs to overcome retrieval problems caused by differences in terminology and the scattering of relevant information across many databases".  There are three parts to the UMLS:  the Metathesaurus, the Semantic Network, and lexical normalization tools. [See Metathesaurus, Semantic Network, lvg]

 
unapproved
Core data is said to be unapproved if it has a status of N, implying that it needs to be looked at by an editor and subsequently approved. [See approved concept]

 
unary integrity check
Refers to an integrity check that requires one additional piece of information.  For example, a check that enforces the rule "do not let <source a> merge with itself if the codes are different" requires one pieces of information, and so is a unary check. These checks make use of the ic_single table in the MID. [See binary integrity check]

 
unreleasable
Core data is said to be unreleasable if it exists in the MID but will not appear in the next version of the Metathesaurus.  Unreleasability is indicated in a core table by a tobereleased value of N or n.  [See releasable]

 
unreviewed
Core data is said to be unreviewed if it has a status of U, implying that it has not been looked at by an editor and will not be looked at by an editor. [See unapproved]

 
update source
An updated version of a source that already exists in the MID. [See obsolete source]



update source insertion
A source insertion involving a new version of a source that has previously been inserted into the MID.  [See new source insertion, partial update insertion, full update insertion]
 
 
versioned source
A source with accompanying version information.  Usually just referred to as a source.


versionless source
Synonym of root source.
 
 
violations vector
An integrity vector indicating which checks failed rather than indicating which checks should be performed.  The integrity system takes an integrity vector as input and gives back a violations vector as output.  In a violations vector the integrity checks are paired with ic states (indicating levels of violation) instead of ic codes (indicating levels of enforcement). [See integrity vector, override vector]

 
worklist
A table containing a list of concepts generated by the WMS.  Worklists exist to facilitate the bulk of the editing and are derived principally from the ME bins.  They are tracked by the EMS throughout their entire life cycle, from assignment to editors, to being checked back in, to being stamped. [See checklist]

 
WMS
The Worklist Management System is a web application designed by the NLM to manage the life cycle of worklists.  The WMS is also used for making checklists. It is available at /cgi-oracle-meowusers/wms.pl  . [See EMS]
