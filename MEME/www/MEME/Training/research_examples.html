Example #1: Report of the Investigation of Missing GOPROC STYs
Laura reported the following example as a GOPROC concept
that should have had a STY after our work to add the
semantic types generated by Steph.  Here is the example:

> CN# 8508953  pyruvate dehydrogenase bypass
> CUI   Concept Status needs Review
> ATOMS
>              N []  pyruvate dehydrogenase bypass
> [GO2002_12_16/PT/GO:0006087]
> (oc_testsrc)

Following is the process used to investigate this.

1. My first suspicion is that there was no STY
   for this atom in the attributes.add file we used.
   When attributes are added via a file like this
   one, typically they are attached in the attributes.src
   file via SOURCE_ATOM_ID id types.  So, what
   we have to do to test this case is these steps:
   a) find the atom_id of this atom. since there is
      only one atom in the concept, we can do this:

SQL> select atom_id from classes where concept_id=8508953;
   ATOM_ID
----------
  10476223

   b) find the source_atom_id of this atom:

SQL> select * from source_id_map where local_row_id=10476223;
LOCAL_ROW_ID TA ORIGIN     SOURCE               SOURCE_ROW_ID
------------ -- ---------- -------------------- -------------
    10476223 C  Local      GO2002_12_16              68033281
    10476223 A  Local      RCD99                      7395805

      Here, we look at the 'C' row, and the SOURCE_ROW_ID
      is the source atom id, 68033281.

   c) Go to the GOPROC* directory and look up this id
      in attributes.add.

oc% cd /d5/MEME/dev/META2004/GOPROC*/Patch
oc% grep 68033281 attributes.src
60150|68033281|C|SEMANTIC_TYPE|Organism Function|GO2002_12_16|N|Y|N|N|SOURCE_ATOM_ID||


      So this confirms that there should be an 'Organism Function' semantic
      type connected to this case.  Laura is right, something
      bad happened.

2. My next thought is that an error occurred while processing
   the attributes file, since if no error had occurred, there would
   be this STY in the actual concept in oc_testsrc.  I looked
   in the log and found this error:

[11-feb-2003 19:00:07] MAP_TO_MEME_IDS: Executing MEME_SOURCE_PROCESSING.map_sg_fields to map source_attributes ...
MEME_ERROR => [11-feb-2003 19:00:11] Error in MEME_SOURCE_PROCESSING.  Method: MAP_SG_FIELDS Location: 350 Error Code: MI0001: S
ystem error. Detail: ORA-01722: invalid number
MEME_ERROR => [11-feb-2003 19:00:11] Error in MEME_SOURCE_PROCESSING.  Method: MAP_SG_FIELDS Location: 350 Error Code: MI0001: S
ystem error. Detail: User-Defined Exception
BEGIN MEME_SOURCE_PROCESSING.map_to_meme_ids (          table_name => 'A',      authority => 'GOPROC2002_12_16',        work_id
=> 2446190 ); END;

*
ERROR at line 1:
ORA-06510: PL/SQL: unhandled user-defined exception
ORA-06512: at "MTH.MEME_SOURCE_PROCESSING", line 1206
ORA-06510: PL/SQL: unhandled user-defined exception
ORA-06512: at "MTH.MEME_SOURCE_PROCESSING", line 5116
ORA-01722: invalid number
ORA-06512: at line 1

   The real error burried in all of this is "invalid number"  the rest
   of it is just filler.  "invalid number" most likely means that
   there is some id type we are using in the attributes file
   where the id field contains non-numeric characters.  To determine
   the exact problem I performed the following steps.

   a) First, I checked to see what id types were being used
       in source_attributes in oc_testsrc

SQL> select distinct sg_type from source_attributes;
SG_TYPE
--------------------------------------------------
SOURCE_ATOM_ID

   b) Given that all are source atom ids, the most likely scenario
      is that there are non-numeric sg_id fields in source_atom_ids.
      This is because in map_to_meme_ids for the SOURCE_ATOM_ID
      id type, a to_number(sg_id) function is applied which throws
      the "invalid number" exception if the sg_id contains non-numeric
      characters.   Since there are not too many rows, I just decided
      to look for objectionable cases by doing this:

SQL> select sg_id from source_attributes;
...
68028002
68028045
biological_process unknown GO:0000004
68028085
68029350
...
68034544
68034547
genetic exchange GO:0009292
68034994
68035007
...

      Amid a series of what looked like source atom_ids were these
      two obviously bad identifiers.

   c) The final step was to verify that these things were actually
      incorrect in the attributes.add file that we used.

apollo% 04
apollo% cd GOPROC*
apollo% grep 'biological_process unknown GO:0000004' attributes.add
62095|biological_process unknown GO:0000004|C|SEMANTIC_TYPE|Biologic Process|GO2002_12_16|N|Y|N|N|SOURCE_ATOM_ID||


Conclusion:
   Laura is correct and the case she cited should have had a
   semantic type.  Two problems lead to this situation.  First
   the data generated had bad sg_id fields, and second the
   processing failed to note that an error had occurred.

Example #2: Report of the Comparison of full and update MSH insertions.
Here, the project was to research examples from both full
and update MSH insertions and determine how they are different.
We started this process by looking up a set of concept_ids containing
status=N current-version MSH atoms.

oa% /site/umls/oracle/scripts/dump_table.pl -u $mu -d oc_testsw -q "select distinct concept_id from classes where source='MSH2003_2003_01_04' and status='N'" >! b
oa% /site/umls/oracle/scripts/dump_table.pl -u $mu -d oc_testsrc -q "select distinct concept_id from classes where source='MSH2003_2003_01_04' and status='N'" >! a

Next we diff'd the files.

oa% diff a b >! diff

We found 83 cases in file a not in file b and zero cases in b not in a.
We chose one (representative) example.

In the full insertion, "Yam" is merged with "Dioscorea". This
is because of the following mergefact:

80116932|SY|80116931|MSH2003_2003_01_04||N|N|MSH2003_2003_01_04|SOURCE_ATOM_ID||SOURCE_ATOM_ID||

Both have the same MUI in MSH.

I checked and in the update insertion, neither of these
2 atoms is inserted, meaning they are both listed in
the "replacement.src" file.

Conclusion:  In the past, an attempt was made to merge these
two cases (because they have the same MUIs) and it was determined
by editors that they should remain separate (hence the concept
level rel between the two concepts):

[REL]  Yam - dietary [MTH/PN||E-TPW|E-CCR] {4750452} C

The merge occurred *again* in the full insertion because
the MUI merge was again indicated, but did not occur in
the update insertion because neither of these records
in MSH changed.   It would appear that the update insertion
handles this both correctly and cleanly.

This error usually  means there is a problem with the RxNorm graph.
Looking at line 566 of RxNormService.java (as the error indicates)
we find this:

563	    if (rnba.getSynonym() != null) {
564	      // If "Bayer Aspirin 100 MG Oral Tablet" just return "Bayer Aspirin 100 MG"
565	      String name = rnba.getSynonym().toString();
566	      cd_wrk.append(name.substring(0,name.lastIndexOf(scd_atom.getDoseForm().toString())-1));
567	    }

This indicates that the SCD connected to the clinical drug with
atom_id 10352240 may have a bad dose form.  We can use SQL
in the database to check this out.

SQL> select atom_id_1,atom_id_2 from relationships where atom_id_1 in
  (select atom_id_2 from relationships where atom_id_1=10352240
    and relationship_name='SFO/LFO') 
  and relationship_attribute='dose_form_of';
 ATOM_ID_1  ATOM_ID_2
---------- ----------
   9956551    9955942

SQL> select * from atoms where atom_id in (9956551,9955942);

   ATOM_ID VERSION_ID
---------- ----------
ATOM_NAME
--------------------------------------------------------------------------------
   9955942          0
Injectable Solution

   9956551          0
Clindamycin 6 MG/ML / Glucose 50 MG/ML Injectable Solution [Cleocin phosphate]

OK, so the SBD and its dose form are correctly matched.  Now, we check
to see if the SY and the dose form match.

SQL> select * from atoms where atom_id in        
  2  (select atom_id_2 from relationships where atom_id_1=9956551
  3   and relationship_name='SFO/LFO');

   ATOM_ID VERSION_ID
---------- ----------
ATOM_NAME
--------------------------------------------------------------------------------
   9917139          0
Cleocin Phosphate Intravenous Solution

So, we have an SBD whose SY has a non-matching dose form.
This probably happened when the "change dose form" utility
from the RxNorm maintenance page was run.  I checked the
utility and it was fixed to run properly, so we need a QA
check added to MID validation script to look for cases of
SYs not matching their respective SBD's dose form.
