MEME_APROCS

This PL/SQL package implements the atomic actions.

aproc_change_field 

    FUNCTION aproc_change_field (
        row_id                  IN INTEGER,
        table_name              IN VARCHAR2,
        field_name              IN VARCHAR2,
        old_value               IN VARCHAR2,
        new_value               IN VARCHAR2,
        action_status           IN VARCHAR2,
        molecule_id             IN INTEGER,
        authority               IN VARCHAR2,
        timestamp               IN DATE
    ) RETURN INTEGER;
This function is the atomic action for changing an arbitrary field  it also recomputes the rank of the row changed if necessary.
aproc_change_concept_id 
    FUNCTION aproc_change_concept_id (
        atom_id                 IN INTEGER,
        old_concept_id          IN INTEGER,
        new_concept_id          IN INTEGER,
        action_status           IN VARCHAR2,
        molecule_id             IN INTEGER,
        authority               IN VARCHAR2,
        timestamp               IN DATE
    ) RETURN INTEGER;
This atomic action is used to change the concept_id of atoms during merges and splits.  It also (implicitly) changes concept_ids for the source level attributes and relationships tied to the atoms being moved.
aproc_change_atom_id 
    FUNCTION aproc_change_atom_id (
        row_id                  IN INTEGER,
        table_name              IN VARCHAR2,
        old_atom_id             IN INTEGER,
        new_atom_id             IN INTEGER,
        action_status           IN VARCHAR2,
        molecule_id             IN INTEGER,
        authority               IN VARCHAR2,
        timestamp               IN DATE
    ) RETURN INTEGER;
This atomic action is used to change the atom_id of a source level relationship or attribute.  Note: it also changes the concept_id corresponding with the atom_id changed.  The table_name cannot be 'C' or 'CS'. 
aproc_change_dead 
    FUNCTION aproc_change_dead (
        table_name              IN VARCHAR2,
        row_id                  IN INTEGER,

        action_status           IN VARCHAR2,
        molecule_id             IN INTEGER,

This is the atomic action to delete rows from core tables.  It is also used by undo to resurrect rows from the dead tables.  It makes use of the bury and digup procedures.

report_insert_data 
    FUNCTION report_insert_data (
        table_name              IN VARCHAR2,
        row_id                  IN INTEGER,
        source_row_id           IN INTEGER,
        authority               IN VARCHAR2,
        molecule_id             IN INTEGER,
        atomic_action_id        IN INTEGER
    ) RETURN INTEGER;
This function will record data from atomic inserts in the operations queue.   The process retrieves records from the input tables, create an XML ActionEvent fragment and stores it as a character large object (CLOB) inside the operations_queue table.
validate_atomic_actions 
    FUNCTION validate_atomic_actions(
        atomic_action_id        IN INTEGER
    ) RETURN INTEGER;
This function validates atomic action. The process accepts atomic_action_id and check records from atomic_actions with that previously processed CRACS atomic_action_id.  It does support actions such as, INSERT, CHANGE_TOBERELEASED, MOVE, CHANGE_STATUS, CHANGE_FIELD and DELETE.  Different validation applies to each of the actions.  When the function returns TRUE, atomic action passed.  When the function returns FALSE, atomic action failed.
connect_concepts 
    PROCEDURE connect_concepts(
      new_value                 IN INTEGER,
      old_value                 IN INTEGER,
      field_name                IN VARCHAR2
   );
This procedure only runs when the MEME_UTILITY.meme_mode is set to MEME_CONSTANTS.MRD_MODE.  When aproc_change_concept_id, aproc_change_atom_id, or aproc_change_field (where field is an atom_id or concept_id) actions are processed in the MRD concepts can become "connected" in a way that the MRD cares about.  This connection is logged in the connected_concepts table.
