Integrity Subsystem

Overview

There are currently two components to the integrity checking system
  • Live Checking . These are the pre and post checks performed when molecular actions are executed.
  • Off Line Checks . These are the checks performed by processes like the matrix initializer and the QA bins part of the EMS.

Live Checking

The integrity checks are now implemented as Java classes in the gov.nih.nlm.meme.integrity package.  All classes are implementations of the IntegrityCheck interface, and each is additionally either an implementation of DataConstraint or MergeInihibitor . Following are some important features of live integrity checks.
  • Called from molecular actions.
  • Constraints can be either reversible or irreversible .  Violations of reversible checks should be used to produce warnings but should not cause an action to fail.  Violations of irreversible checks should cause actions to fail.
  • Live checking can be done either before or after an action takes place.  For actions like approve concept, checking should be done before approval to ensure that the data is in the correct state.  For actions, like merge, checking should be done before merging to ensure that the merged concept will have no violations.  For all actions, data constraints should be checked to inform editors of potential violations in the state of the concepts being edited.
  • An EnforcableIntegrityVector is used to keep track of which checks are being used.  Enforceable integrity vectors can be obtained from the MEMEDataSource .
  • For editors of a certain level, the default checking scheme in an enforcable integrity vector can be overridden by obtaining an override vector from the data source ( MEMDataSource.getOverrideVector )  and applying it to the enforcable vector (using the applyOverrideVector method).
  • Some checks will require both the source and target concepts to be checked together, others will look at just a single concept..
  • Checking is performed by calling one of the EnforcableIntegrityVector  methods.  The return value of this call is a ViolationsVector .

Offline Checks

The other side of the integrity system is implemented in PL/SQL and using some key procedures can calculate the integrity state for either the whole database, or the subset that has been edited since the last time the procedure was run.

MEME_INTEGRITY.matrix_initializer Procedure

  • Sets releasability/status of concepts.
  • Calculates editing_matrix values.
  • Calculates integrity state for all concepts (initializer), or subset of concepts (updater).  The updater as several modes:
    • It can calculate the integrity for a subset of concepts
    • It can calculate the integrity state for a set of concepts touched since a certain time
    • It can "catch up", which involves calculating integrity state for all concepts touched since the last time the initializer/updater was run.
  • Logs its activity (max_tab, is_handled_actions, max_action_tab, etc.)
MEME_INTEGRITY.actions_nitializer Procedure
  • Called by the Matrix Initializer.
  • Creates change status action rows for those concepts whose status is being changed by the initializer.
MEME_INTEGRITY.violations_nitializer Procedure
  • Initializes IC_Violations with violations found by the Matrix_Initializer.
  • This table is only really used by the integrity_server which is not currently part of the MEME design, so it is currently unimplemented.

Creating a New Integrity Check.

When creating a new integrity check, or modifying old ones, there are a number of things which must be done in order for that new check to work within the integrity system.  Following are the steps involved.
  • Create the new check.  This involves writing a new implementation of either DataConstraint or MergeInhibitor, depending opon whether the check requires a single concept or a pair of concepts.  The various abstract classes in the gov.nih.nlm.meme.integrity package may be helpful starting points.
  • Update integrity tables.  There are a number of important integrity tables which need to be updated with this new information.
    • integrity_constraints .   This table tracks the following information: ic_name, violation actions, corrrection actions, status (Active, or I nactive), type (Reversible, or Irreversible), activation date, deactivation date, short description, long description.
    • ic_applications .  In this table, the application integrity vectors need to be updated to reflect the new constraint.  To do this, use the integrity editor .
    • ic_override . In this table, the override vectors need to be updated to reflect the new constraint.  To do this, use the integrity editor .
    Make sure that these changes are made in all appropriate databases.
  • Matrix Initializer changes.  If the check is to be performed by the matrix initializer then it needs to be implemented in PL/SQL as well to operate on the whole database or a subset and added to the MEME_INTEGRITY_PROC package.
  • Integrity Snapshot changes.  Where appropriate, database-wide queries for this constraint (typicall for new MGV  constraints) should be created and added to the MEME_SNAPSHOT_PROC package as procedures called snapshot_<ic_name>.  This procedure must implement pre/post checking as well as the diff logic, look at the code for one of the other checks as a guide for how to do this.
  • Update documentation.  This page should be updated to reflect the changes/additions.

    Using the Integrity System

    There are things that an application must do in order to make user of the integrity system.  Since MEME4 is a client/server enviornmnent, most of the actual interaction with the integrity system will take place on the server side.  Client applications need only use a MEMETicket object that is properly configured with the application name and editor level..  For example,
      MEMETicket ticket = new MEMETicket();
      ticket.setApplication("EDITING_INTERFACE");
      ticket.setEditorLevel(0);
    On the server side, any applications that need to use the integrity system will probably require that the client pass this kind of ticket information.  This can then be used to request vectors from the data source object. For example,
      MEMETicket ticket = ... obtain ticket from client...;
      MEMEDataSource mid = .... obtain data source ....;
      IntegrityVector vector = mid.getEnforcableIntegrityVector(ticket.getAuthority());
      IntegrityVector override_vector = mid.getOverrideVector(ticket.getEditorLevel());
      vector.applyOverrideVector(override_vector);
      Concept concept = .... obtain a concept ... ;
      ticket.setViolationsVector(vector.applyDataConstraints(concept));
    This code would appear on the server side.  It obtains the ticket, the data source and a concept.  It retrieves the necessary vectors applies the override vector to the vector and then applies the vector to a concept.  The result is a ViolationsVector which is stored in the ticket.  This could then be returned to the client who would show it to the user.
