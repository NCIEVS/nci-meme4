RRF Source Data Proposal - DRAFT (02/17/2005)

 

Overview/Objective

The RRF is the standard release format for the UMLS Metathesaurus. We would like to push its use as a standard submission format for source providers whose terminologies will be added/updated to the Metathesaurus. We need to preserve the fundamental semantics off the RRF so that it can be easily and cleanly documented and explained.

This document details how each file should be used as an input file.

Proposal

Following is a field-by-file proposal for how the RRF will serve as a source insertion format and how each of the data fields will be interpreted by our insertion formats. Some important things we discuss are:
  • Source Data Representation
    • terms
    • attributes
    • relationships
  • Representing Map Sets
  • Suggesting UMLS Synonymy
  • Suggesting Default Semantic Types
  • Representing Names and Identifiers Properly
    • Creating 7-bit forms
    • Avoiding compound inflected forms (e.g. "dog(s)")
    • Use meaningless unique identifiers
  • Choosing Term Types
  • Representing Sources as Concepts
  • Tools
  • Sort Order & Uniqueness
.

Source Data Representation

Before we start, there are a few important concepts to understand:

"proxy ui"
A line-level identifier in a file indented to be a value for joining to other files. Each of the Metathesaurus UI fields CUI, AUI, RUI, ATUI should be populated with proxy uis. They serve as unique identifiers for their respective files and when they must have unique values across all files.  In the RRF format as released in the UMLS contains NLM computed identifiers for these fields.  Since source providers are not expected to have access to NLM computed identifiers

"proxy atom ui"
Must appear in AUI field in MRCONSO.RRF, one unique value per line. If the source asserts SAUI values, use them here.

"proxy concept ui"
Must appear in CUI field in MRCONSO.RRF, one unique value per source concept. If source asserts SCUI values, use them here.

"proxy attribute ui":
Must appear in any available ATUI field. If source asserts SATUI values, use them here. Currently, these values are not used but may be in the future

"proxy relationship ui":
Must appear in RUI field in MRREL.RRF, if source asserts SRUI values, use them here.

The following section describes each RRF file in general terms. It indicates how the data from this file will be used in the MID by the source insertion process. Next, a field-by-field look is taken to understand what data a source provider should use in each case and exactly how it will be used by the source insertion process. Finally, examples are provided. Note, many RRF files will not need to be populated when representing source data.

MRCONSO.RRF

This is the most important of the files and contains the actual terms from the source as well as its "conceptness". The most important thing to understand about this file is the use of "proxy atom ui" and "proxy concept ui".   We highly recommend that source providers populate the CODE, SCUI and SAUI fields.  We also highly recommend that to avoid confusion the values for CUI and AUI be the same ones as used in SCUI and SAUI.  While this may seem redundant it is not because semantically CUI,AUI are used as join fields to the other files, whereas SCUI and SAUI are interepreted to be the concept and atom id values for atoms of this source.  There is no requirement that they use the same value ranges, but it may help to eliminate confusion.

For source insertion purposes, this data will be loaded into classes and concept_status. Also the index tables will be populated.  Rows with the same CUI values will be used to perform merge sets (by loading data into mom_precomputed_facts).

MRHIER.RRF

Source hierarchies are best represented here. MRHIER.RRF rows should be attached to atoms via "proxy atom ui" values.  Each relationship represented here may also be represented as PAR/CHD relationships in MRREL (although that information would likely be ignored in favor of this data).  Source providers may choose to represent hierarchical information ONLY as PAR/CHD relationships in MRREL, in which case this file should be left empty.

For source insertion purposes, this data will be loaded into context_relationships.  SIB rels will likely be created for all sources that have a SIB count smaller than some threshold relative to total source size.  e.g. < 10*PAR/CHD count.  If this file is empty, we will build hierarchies from PAR/CHD rels in MRREL.

MRREL.RRF

This is where source relationships are stored. Relationships may exist an any level in a source but must be assigned a "proxy atom ui" to represent the class level at which they are connected. The STYPE fields indicate which MRCONSO field is actually the level at which relationships are connected. 

PAR/CHD/SIB relationships can be used here to represent hierarchical information.  Hierarchies can be more unambiguously represented in MRHIER.RRF, but source providers may choose to only represent this information in the form of distance-one PAR/CHD and SIB relationships.

Important note: in the UMLS, relationships are read from right to left, so the CUI2 is REL to CUI1.  Thus if REL is RN, it means that CUI2 "is narrower than" CUI1 and not the other way around.  Relationships may be represented in one direction or in both directions.

Synonymy with Metathesaurus concepts should be represented with REL=SY relationships where the CUI2 is a UMLS CUI and the STYPE2=UMLSCUI and the AUI2 is blank.  Synonymy with Metathesaurus atoms should be represented with REL=SY relationships where AUI2 is a UMLS AUI and the STYPE2=UMLSAUI and the CUI2 is the CUI of that AUI value in the actual Metathesaurus.

For source insertion purposes, this data will be loaded into relationships.  Suggested merge facts may also appear here in the form of SY relationships.  Any SY relationship should result in a merge fact, whether it merges a SAUI from this source to an AUI or CUI in the Metathesaurus or if it merges atoms within this source.  PAR,CHD,SIB relationships will likely be ignored.

  How should relationships connected to arbitrary classes of identifiers be represented? 

  

This
issue is about representing something like a SNOMEDCT relationship in
the MRREL format.  The UMLS always connects relationships at the
atom (AUI) level, meaning that a row of MRREL represents a relationship
between two AUIs.  The STYPE fields are used to indicate the level
in MRCONSO at which the source asserted the relationship.  For
example, a SNOMEDCT relationship would use SCUI as the STYPE values,
indicating that a SNOMEDCT relationship is asserted as being between
two SNOMEDCT "concepts".  In MSH, the STYPEs would be "SDUI" in
many cases, indicating a relationship between one descriptor class and
another.  In MRREL as an input format, we should maintain this
methodology with one possible exception, which is really a
shortcut.  We should allow CUI1 to be populated, AUI1 to be blank,
and STYPE1=SCUI to indicate that the source is asserting a relationship
between two concepts in that source, this is a natural extension of the
way "concept level" relationships are asserted in the Metathesaurus
MRREL file.

  

The alternative methodology is to populate the AUI with any kind of identifier and use STYPE to indicate the type of identifier.

  How should attributes connected to arbitrary classes of identifiers be represented? 

  

This
issue is about representing something like a SNOMEDCT CONCEPTSTATUS
attribute in
the MRSAT format.  The UMLS always connects attributes to core
Metathesaurus UIs (AUI, RUI).  This means that a row of MRSAT is
an attribute connected to an atom or a relationship. The STYPE field is
used to indicate the level
in MRCONSO (or MRREL) at which the source asserted the attribute. 
For
example, a SNOMEDCT CONCEPTSTATUS attribute would use SCUI as the STYPE
values,
indicating that a SNOMEDCT CONCEPTSTATUS attribute is asserted as being
attached to a SNOMEDCT "concept".  In MSH, the STYPE would be
"SDUI" in
many cases, indicating attributes attached to MSH descriptor
classes.  In MRSAT as an input format, we should maintain this
methodology with one possible exception, which is really a
shortcut.  We should allow CUI to be populated, METAUI  to be
blank,
and STYPE=SCUI to indicate that the source is asserting a "concept
level" attribute. This is a natural extension of the
way "concept level" relationships are asserted in the Metathesaurus
MRREL file.


  

The alternative methodology is to populate the METAUI with any kind of
identifier and use STYPE to indicate the type of identifier.  Thus
METAUI could be a CODE, an SCUI, an SDUI, or any other value rather
than being an AUI/RUI as it is in the output format.

source providers can supply us with suggestions. We can release tools
to help build defaults from their hierarchy  information

Suggesting UMLS Synonymy

In many cases, sources might wish to explicitly suggest synonymy with UMLS CUIs. This suggested synonymy can be indicated by supplying MRREL rows with the following characteristics:
  • REL=SY
  • AUI1="proxy atom ui" representing the source concept to merge with UMLS CUI
  • CUI2=suggested synonymous UMLS CUI value
  • STYPE2=CUI
During the source insertion process, these relationships will be represented as SY relationships. A merge facts set will always be performed to attempt to merge across a source's SY facts.
 .

Suggesting Default Semantic Types

Sources should be encouraged to submit their own default semantic type suggestions. This can be done by using the MRSTY file, since sources do not natively contain UMLS semantic type information. One of the tools that can be built and supplied with the RRF documentation/training is one that makes use of the hierarchies (like we do in inversion) to create and inherit default STYs down the hierarchies.
 

Representing Names and Identifiers Properly

All information in the files (especially within the concept names) should use the same character set.  UTF-8 is preferred, but as long as a standard character set is used, conversion to UTF-8 for inclusion in the Metathesaurus should be trivial.   For all names that contain non-ASCII characters, a pure 7-bit ASCII form of the name should be provided and connected to the original form via an SY relationship (with a RELA of 'plain_text_form_of').

Avoid using compound strings like "dog(s)".  Instead have a PT with the preferred form, like "dog" and use SY atoms for the variants, like "dogs".

Avoid using leading or trailing spaces (e.g. " Apple" or "Apple ").  Avoid names with multiple spaces (e.g. "this  has multiple     spaces").

Investigate cases of names that use characters with very low character frequency counts.

Use meaningless unique identifiers for the SCUI, SAUI, SRUI, and SATUI fields that carry no inherent semantic information.  For example, a certain SAUI value should not imply that a term is, say, a clinical drug, or a procedure..

Choosing Term Types

One big challenge for users of the RRF as a source submission format will be to use TTYs properly. We should use the previous release MRDOC as the gold standard and train users to choose from the available set with certain ones (like PT, SY) being preferred over others (like MH, or N1). Generally, source providers should not be allowed to create entirely NEW TTY values. NLM should also reserve the right to override source provider TTY suggestions to avoid conflicts.

Representing Sources as Concepts

Each new source will require a new SRC concept. Most of this information can come from the MRSAB.RRF file. We should require that certain MRSAB.RRF fields are populated (like source official name, source short name, etc). We should also standardize the use of SRC/RHT to represent the context tree-top.  Should we encourage/require source providers to actually supply SRC concepts that conform with the spec?

Tools

One of the key selling points for using the RRF as a format for representing source data is that we can make available a suite of tools for improving the quality of the source vocabulary.  Some tools may include
1. QA tools.  This would include UI referential integrity tools for verifying that the format is being properly represented as well as semantic rules like "each PT should have a tree position in the hierarchy".
2. MRFILES/MRCOLS building tools.
3. Hierarchy building  tools.  Source providers can start with PAR/CHD lists or code ranges and tools will automaticlaly build full MRHIER representations of the hierarchy.
4. STY defaulting tools.  Given a hierarchy and assignments of default semantic types for higher level terms in a hierarchy, this tool would create semantic type defaults for all descendents in the hierarchy.
5. Building 7-bit forms.  Standard tools exist for transliterating non-ASCII UTF-8 strings into 7-bit ASCII forms.  We could also verify the other rules about strings (no trailing/leading spaces or junk characters).
 

Sort Order

All files should be sorted using a byte-order sorting.  This conforms to a UNIX sort with LANG=C.  This way, we can guarantee source transparency, including the sort order of original files.  No file should have duplicate lines within it, each line should be unique.
