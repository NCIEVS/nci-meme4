MEME3 PL/SQL API

This document contains descriptions/APIs for the key MEME3 objects, including PLSQL packages and OpenROAD applications.  The components described in this document are:
  • MEME_* PLSQL Package API
  • MEME_CONSTANTS
  • MEME_UTILITY
  • MEME_SYSTEM
  • MEME_RANKS
  • MEME_APROCS
  • MEME_INTEGRITY
  • MEME_INTEGRITY_PROC
  • MEME_BATCH_ACTIONS
  • MEME_SOURCE_PROCESSING
  • MEME_TESTER
  • MEME_DUMP
Due to package dependencies, they must be loaded in the following order,
  • MEME_CONSTANTS
  • MEME_UTILITY
  • MEME_SYSTEM
  • MEME_RANKS
  • MEME_APROCS
  • MEME_BATCH_ACTIONS
  • MEME_INTEGRITY_PROC
  • MEME_INTEGRITY
  • MEME_SOURCE_PROCESSING

The MEME_* PLSQL Package API [next]

Each of the PLSQL packages used by the MEME system employs a standard set of variables, functions, and procedures.  Any PLSQL packages added to the system should conform to this API so that any users of the packages can expect to get at this information.

package_name 
 package_name    VARCHAR2(25);
The name of the package.  For example  MEME_RANKS.package_name is equal to 'MEME_RANKS'.
release_number 
 release_number  VARCHAR2(1);
This is the current release number of the MEME system.  As of 1/20/2000 the current release is 3 (MEME3).
version_number 
 version_number  VARCHAR2(5);
The current version number within the current release.
version_date 
 version_date        DATE;
This is the release date of this particular release and version.
version_authority 
 version_authority   VARCHAR2(3);
Initials of the person last responsible for the release.
location 
 location                VARCHAR2(5);
This variable is used by the procedures/functions to track how far along the procedure has gotten (if an error is produced).  This should be declared at the package level so that it is not necessary for each procedure to employ a local location number.
err_msg 
 err_msg         VARCHAR2(256);
Like location, this variable is used by the procedures/functions to prepare error messages for potential problems.  A typical scenario is before doing some operation the location and err_msg are set.  If the operation throws an exception, the EXCEPTION section of the procedure can use the location and err_msg variables to correctly report the error.
<package_name>_debug, <package_name>_trace 
 <package_name>_debug    BOOLEAN := FALSE;
 <package_name>_trace    BOOLEAN := TRUE;
<package_name> should be substituted for the actual package name.  Later we will see functions/procedures which use these boolean variables.  If the _debug one is true then SQL statements should not actually be executed.  If the _trace flag is true, then SQL statements should be printed to the screen.  See the local_exec function below.
<package_name>_exception 
 <package_name>_exception EXCEPTION;

Every package should declare a package level exception which its

release, version 
    FUNCTION release
    RETURN INTEGER;

    FUNCTION version
    RETURN FLOAT;

    PROCEDURE version;
The release function returns the release number.  The version function returns the version number and the version procedure just prints the version number to the screen.
version_info 
    FUNCTION version_info
    RETURN VARCHAR2;
    PRAGMA restrict_references (version_info,WNDS,RNDS,WNPS);
This function returns an informative message containing all of the version information.  It can be used in an SQL statement like:
select meme_ranks.version_info from dual;
This would return something like this (in SQL*Plus):
VERSION_INFO

MEME_RANKS Release 3: version 2.1 (18-JAN-00)
set_trace_on, set_trace_off, set_debug_on, set_debug_off 
    PROCEDURE set_trace_on;
    PROCEDURE set_trace_off;
    PROCEDURE set_debug_on;
    PROCEDURE set_debug_off;
These functions toggle the trace and debug flags describe above.
trace 
    PROCEDURE trace ( message IN VARCHAR2 );
This function prints a message to the screen ONLY if the trace flag is TRUE.  This is good for debugging instead of put_message because turning the flag off prevents messages from being printed (instead of having to remove calls to MEME_UTILITY.put_message).
local_exec 
    PROCEDURE local_exec (query in varchar2);
    FUNCTION local_exec (query IN VARCHAR2) RETURN INTEGER;
local_exec uses the trace and debug flags.  If the trace flag is on, it prints the query to the screen.  If the debug flag is on it calls MEME_UTILITY.exec.  The function calls the MEME_UTILITY function version of exec, and the procedure calls the procedure version.
help 
    PROCEDURE help;

    PROCEDURE help ( 
        topic IN VARCHAR2
    );
These procedures are used for obtaining help about a package.  The version without parameters returns a general help message indicating which topics are available.  The other version looks up the topic in the application_help table and returns the help.
self_test 
    PROCEDURE self_test;
In most packages this is currently unimplemented, but eventually packages should be able to run a self-diagnostic to ensure correct functionality of the member functions.  This may be deprecated by the MEME_TESTER package.
<package_name>_error 
    PROCEDURE <package_name>_error (
        method              IN VARCHAR2,
        location            IN VARCHAR2,
        error_code          IN INTEGER,
        detail              IN VARCHAR2
    );
This function is used by the functions/procedures of the package to report errors.  The method is the method of the package, location is typically the value of the location variable, error_code is a code which eventually should be understood by MEME_CONSTANTS, and detail is a detailed description of the problem.  Typically this procedure should call MEME_UTILITY.put_error.

MEME_RANKS [previous/next]

This PLSQL package contains procedures and functions for ranking various things in the MEME system.  Additionally, the ranking functions for fields can serve as validation functions, returning empty ranks or throwing exceptions if the value is not valid.

get_rank 
    FUNCTION get_rank (
        row_id          IN INTEGER,
        table_name      IN VARCHAR2
    ) RETURN INTEGER;
This function takes a row_id and the name of a table and returns the integer rank of that row in that table.  It employs some of the following methods.
set_rank 
    FUNCTION set_rank (
        row_id          IN INTEGER,
        table_name      IN VARCHAR2,
        field_name      IN VARCHAR2 := NO_FIELDS_CHANGED
    ) RETURN INTEGER;
This function sets the rank of a particular row in a particular table.  It takes this optional field_name parameter to indicate that a field was recently changed.  This procedure is employed by MEME_BATCH_ACTIONS after changing a field.  If changing the field for that particular table does not require a rank recomputation, the rank will NOT be recomputed..
rank_to_number 
    FUNCTION rank_to_number (
        rank    VARCHAR2
    ) RETURN INTEGER;
Most of the ranking functions in this package return VARCHAR2  ranks which are too long to be practical as integers (because they include the row_id).  This function takes a VARCHAR rank, strips off the row_id part and returns an integer representing just the rank part.  This is the number stored in the rank field of the core tables.This function can be executed in SQL queries.
set_preferred_id, calculate_preferred_atom, get_preferred_atom. 
    FUNCTION set_preferred_id (
        concept_id      IN INTEGER,
        changed_field   IN VARCHAR2 := NO_FIELDS_CHANGED
    ) RETURN INTEGER;
    FUNCTION calculate_preferred_atom (
        concept_id IN INTEGER
    ) RETURN INTEGER;
The first function takes a concept_id and a potentially changed field and determines if the preferred atom_id of the concept needs to be recomputed.  If so, it is recomputed and set in concept_status. This method is also used by MEME_BATCH_ACTIONS. set_preferred_id uses calculate_preferred_atom to determine what the preferred_atom_id for the concept should be.  Calculate_preferred_atom can be used in SQL statements like:

generate_facts 
    PROCEDURE GENERATE_FACTS(
        termgroup_table_1       IN VARCHAR2,
        termgroup_table_2       IN VARCHAR2,
        merge_set               IN VARCHAR2,
        string_flag             IN VARCHAR2,
        code_flag               IN VARCHAR2,
        table_name              IN VARCHAR2,
        source                  IN VARCHAR2,
        authority               IN VARCHAR2);
This procedure is used to generate merge facts for pre-insert merges as well as real merge sets.  It takes two tables of termgroups, the first corresponding to the new ones, the second corresponding to the existing ones.  The string_flag parameter must be in ('NORM','BOTH','EXACT','NOT','NONE'). The code_flag must be in ('SAME','DIFF','NONE').  The table_name is the source of the facts and must be in ('RELATIONSHIPS','CLASSES','SOURCE_CLASSES_ATOMS').  Using these parameters, generate_facts dynamically produces queries and creates the table mom_candidate_facts.
filter_facts

        type_1                  IN VARCHAR2,

        authority               IN VARCHAR2,
        source                  IN VARCHAR2);
This proceure is largely unimplemented but will eventually allow for a wide variety of filtering predicates.  It operates on the mom_candidate_facts removing those that meet the filtering criteria.  Currently the procedure allows for 4 types of filtering (1) EXCLUDE_LIST (where termgroups are on mom_exclude_list), (2) NORM_EXCLUDE_LIST (where termgroups are on mom_norm_exclude_list), and (3) SAFE_REPLACEMENT (where atoms are safe replacements), (4) NONE (no filtering).  The filtering for type 1 and type 2 need not be the same.  for example, if type_1 is NONE and type_2 is EXCLUDE_LIST, any facts where the second termgroup is on the exclude list will be deleted. replacement_merges 
    PROCEDURE REPLACEMENT_MERGES(
        fact_table              IN VARCHAR2,
        merge_set               IN VARCHAR2,
        normalization_flag      IN VARCHAR2,
        ambig_flag              IN VARCHAR2,
        authority               IN VARCHAR2);
This procedure implements a merging algorithm which sets concept_ids in source_classes_atoms to correspond with concept_ids in classes, and unsets the source_concept_status switches for concepts that no longer need to be inserted.  The normalization_flag allows for merging of higher concept_ids into lower ones or vice versa.  Use MEME_CONSTANTS.HIGHER_INTO_LOWER, or MEME_CONSTANTS.LOWER_INTO_HIGHER.  The ambig_flag allows you to require that merges take place only if the facts are 1-1, so 2 new concepts cannot merge into the same existing one.
safe_replacement 
    PROCEDURE SAFE_REPLACEMENT(
        string_flag             IN VARCHAR2 := 'NONE',
        code_flag               IN VARCHAR2 := 'ANY',
        old_source_table        IN VARCHAR2 := '',
        new_source_table        IN VARCHAR2 := '',
        old_termgroup_table     IN VARCHAR2 := '',
        new_termgroup_table     IN VARCHAR2 := '',
        mode_flag               IN VARCHAR2 := 'CREATE',
        change_status           IN VARCHAR2 := 'N',
        source                  IN VARCHAR2,
        authority               IN VARCHAR2);
This procedure dynamically creates queries to perfom safe replacment operations.  The string flag must have values in ('NONE','LUI','SUI').  The code flag must have values in ('ANY','SAME').  If safe replacement is based on source names the old_source_table, and new_source_table parameters should point to tables with source fields.  If safe replacment is based on termgroups, the old_termgroup_table and new_termgroup_table parameters should point to tables containing a single termgroup field.  The mode_flag is either 'CREATE' meaning create a mom_replace_<source> table, or 'INSERT' meaning insert into an existing one.  The 'INSERT' option is for sources which have complicated, multi-step replacement algorithms.  The change_status flag indicates whethr or not the status of the replaced atoms should be changed.  'N' is used for multi-part safe replacement algorithms where you do not want to change the status until the whole safe replacement table has been generated.
setup_partitions 
    PROCEDURE SETUP_PARTITIONS;
This procedure is unimplemented, but when we are using Oracle 8i, it should be called at the beginning of the source insertion process to create partitions in the core tables big enough to hold all of the new rows.  That way, editing can proceed while source insertion is taking place because the new and existing rows will be in different partitions.  One requirement might be that when replacing a source, all atoms from that source must be put into unaccessible partitions, otherwise pre-insert merge sets might have unstable concept ids.
