Automated QA


Overview
We have developed an automated QA system that tracks snapsots of insertion data and mid data. An set of adjustment tables can be used to account for differences between source-mid data and between mid-mid snapshots. The system is maintained by several tables as well as some PL/SQL procedures in the MEME_INTEGRITY package.  There is a web-based front end to make use of this code called the Automated QA System.

Following is a description of the tables used by the automated QA system.
 

Table Fields Description
src_qa_results
 
 qa_id     NUMBER(12) NOT NULL,
 name      VARCHAR2(100) NOT NULL,
 value     VARCHAR2(100),
 qa_count  NUMBER(12) NOT NULL,
 timestamp DATE NOT NULL
 This table holds results of counts generated from the source_* tables. These counts are generated by MEME_INTEGRITY.src_monster_qa. The procedure generates new data with the same qa_id as the other rows. Data for old sources must be removed from this table in a different way.
mid_qa_results
 
 qa_id     NUMBER(12) NOT NULL,
 name      VARCHAR2(100) NOT NULL,
 value     VARCHAR2(100),
 qa_count  NUMBER(12) NOT NULL,
 timestamp DATE NOT NULL
 This table holds the results of counts generated from the core tables. These counts are generated by MEME_INTEGRITY.monster_qa. Each time the monster_qa procedure is run, this table's results are copied to mid_qa_history and it is truncated. Results with a new qa_id are loaded.
mid_qa_history
  Same as mid_qa_results This table holds results from past monster_qa runs. It is used in mid-mid snapshot comparisons (MEME_INTEGRITY.mid_mid_qa_diff).
mid_qa_queries
 
 name      VARCHAR2(100) NOT NULL,
 query     VARCHAR2(1000) NOT NULL
 This table holds the queries used by MEME_INTEGRITY.monster_qa to generate mid snapshot counts. It contains queries not found in src_qa_queries. 
src_qa_queries
 
 name      VARCHAR2(100) NOT NULL,
 query     VARCHAR2(1000) NOT NULL
 This table holds the queries used by MEME_INTEGRITY.src_monster_qa to generate counts from the source tables. Each query in this table should have a comparable query in mid_qa_queries. The editing database (oa_mid2003) should contain the authoritative set of queries.
qa_adjustment
 
 qa_id     NUMBER(12) NOT NULL,
 name      VARCHAR2(100) NOT NULL,
 value     VARCHAR2(100),
 qa_count  NUMBER(12) NOT NULL,
 timestamp DATE NOT NULL,
 description      VARCHAR2(1000)
 This table holds adjustment counts. It is used when comparing src-mid results. The qa_id should match the qa_id in src_qa_results. The count in this table is added to the count for the matching name/value in src_qa_results. The description field should be used to indicate why the change happened.
qa_diff_adjustment
 
 qa_id_1   NUMBER(12) NOT NULL,
 qa_id_2   NUMBER(12) NOT NULL,
 name      VARCHAR2(100) NOT NULL,
 value     VARCHAR2(100),
 qa_count  NUMBER(12) NOT NULL,
 timestamp DATE NOT NULL,
 description      VARCHAR2(1000)
 This table is used to account for differences between two mid snapshots. The qa_count is directional, it is added to the count for the matching name/value for qa_id_1 and subtracted from the matching count for qa_id_2.
qa_diff_results
 
 name      VARCHAR2(100) NOT NULL,
 value     VARCHAR2(100),
 qa_id_1   NUMBER(12) NOT NULL,
 count_1   NUMBER(12) NOT NULL,
 qa_id_2   NUMBER(12) NOT NULL,
 count_2   NUMBER(12) NOT NULL,
 timestamp DATE NOT NULL
 This table is the result of running either src_mid_qa_diff or running mid_mid_qa_diff. It tracks all differences after adjustments have been accounted for. This data is not maintained Each time new diffs are generated the table is truncated.

Following is a description of the procedures used by the automated QA system.
 

Procedure Description
MEME_INTEGRITY.monster_qa This procedure moves mid_qa_results to mid_qa_history and then truncates the mid_qa_results table. It computes the counts from the queries in mid_qa_queries and loads the results into mid_qa_results.
MEME_INTEGRITY.src_monster_qa This procedure looks up the qa_id in src_qa_results. It then computes counts based on the src_qa_queries table and loads these results into src_qa_results. note: the table is not truncated, only additional data is added. In the future, some mechanism for removing data pertaining to old sources should be added.
MEME_INTEGRITY.src_mid_qa_diff This procedure compares the src_qa_results and mid_qa_results tables. Adjustment counts found in qa_adjustment are added to the counts found in src_qa_results. If any discrepancies in counts exist for matching name,value pairs these are recorded in the qa_diff_results table.
MEME_INTEGRITY.mid_mid_qa_diff This procedure compares the current mid_qa_results and the rows from mid_qa_history where the qa_id matches the history_qa_id parameter. Adjustment counts found in qa_diff_adjustment are used to account for differences (adjustments are added to the qa_id_1 results from qa_diff_adjustment and subtracted from the qa_id_2 results). Any discrepancies in counts after adjustments are accounted for are reported in qa_diff_results.
