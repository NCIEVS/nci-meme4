MEME_SOURCE_PROCESSING


This PLSQL package is for inserting sources. There are two template scripts for inserting sources that heavily leverage these procedures: $MEME_HOME/bin/source.csh for english language sources, and $MEME_HOME/Patch/insert_foreign.csh for foreign language sources. 
 

assign_atuis 
    PROCEDURE assign_atuis(
        authority           IN VARCHAR,
        work_id             IN INTEGER,
        table_name          IN VARCHAR := 'SA'
    );
This procedure takes an authority and using attributes_ui it assigns new attribute unique identifiers (ATUI) to source_attributes, and inserts those new rows into attributes_ui.
assign_auis 
    PROCEDURE assign_auis(
        authority           IN VARCHAR,
        work_id             IN INTEGER,
        table_name          IN VARCHAR := 'SC'
    );
This procedure takes an authority and using atoms_ui it assigns new atom unique identifiers (AUI) to source_classes_atoms, and inserts those new rows into atoms_ui.  This procedure works for both english and foreign sources.
assign_string_uis 
    PROCEDURE assign_string_uis(
        authority           IN VARCHAR,
        work_id             IN INTEGER
    );
This procedure takes an authority and using string_ui, source_string_ui, and source_classes_atoms it assigns new string_ids (SUI, LUI, ISUI) to source_classes_atoms, and inserts those new rows into string_ui.  This procedure works for both english and foreign sources.
assign_source_ids 
    PROCEDURE assign_source_ids(

map_to_meme_ids 
    PROCEDURE map_to_meme_ids(
        table_name          IN VARCHAR2,
        authority           IN VARCHAR2,
        unique_flag         IN VARCHAR2,
        work_id             IN INTEGER
    );
This procedure maps any id type found in code_map where table_name='sg_type'  in the source_relationships or source_attributes tables into atom_ids and concept_ids. The source_attributes and source_relationships cannot be mapped until source_classes_atoms has been inserted into source_id_map AND classes.  This procedure makes use of map_sg_fields.

safe_replacement 
    PROCEDURE safe_replacement(
        string_parameter    IN VARCHAR2 := 'NONE',
        code_parameter      IN VARCHAR2 := 'NONE',
        old_source_table    IN VARCHAR2 := '',
        new_source_table    IN VARCHAR2 := '',
        old_termgroup_table IN VARCHAR2 := '',
        new_termgroup_table IN VARCHAR2 := '',
        change_status       IN VARCHAR2 := MEME_CONSTANTS.NO,
        source              IN VARCHAR2,
        authority           IN VARCHAR2,
        work_id             IN INTEGER
    );
This procedure dynamically creates queries to perfom safe replacement operations.  The string_parameter must have values in ('NONE', 'EXACT', 'NORM', 'NOT').  The code_parameter must have values in ('NONE','EXACT','NOT').  If safe replacement is based on source names the old_source_table, and new_source_table parameters should point to tables with source fields.  If safe replacement is based on termgroups, the old_termgroup_table and new_termgroup_table parameters should point to tables containing a single termgroup field.  The change_status flag indicates whether or not the status of the replaced atoms should be changed.  For sources that have multi-part, complex safe replacement algorithms, use multiple calls to this procedure.  If a mistake is made, use reset_safe_replacement to undo all of the steps.  It calls MEME_BATCH_ACTIONS.
reset_safe_replacement 
    PROCEDURE reset_safe_replacement(
        source              IN VARCHAR2,
        authority           IN VARCHAR2);
This removes rows from mom_safe_replacement and sr_predicate which match the source parameter.. 
source_replacement 
    PROCEDURE source_replacement(
        old_source          IN VARCHAR2,
        new_source          IN VARCHAR2,
        authority           IN VARCHAR2,
        work_id             IN INTEGER
    );
This procedure is the key to an "efficient update" style source insertion recipe.  It relies on the existence of a replacement.src file which contains the identifiers from a obsolete version of the source which are essientially identical in the current version.  The procedure goes through and updates that content so that it appears to come from the current version.  It also makes all old-version context relationships obsolete (since the entire context tree must be replaced).
generate_facts 
    PROCEDURE generate_facts(
        termgroup_table_1   IN VARCHAR2,
        termgroup_table_2   IN VARCHAR2,
        merge_set           IN VARCHAR2,
        string_parameter    IN VARCHAR2,
        code_parameter      IN VARCHAR2,
        table_name          IN VARCHAR2,
        source              IN VARCHAR2,
        authority           IN VARCHAR2,
        work_id             IN INTEGER
    );
This procedure is used to generate merge facts for pre-insert merges as well as real merge sets.  It takes two tables of termgroups, the first correspond to the new ones, the second correspond to the existing ones.  The string_parameter must be in ('NORM','EXACT','BOTH','NONE','NOT'). The code_parameter must be in ('EXACT','NOT','NONE').  The table_name is the source of the facts and must be in ('RELATIONSHIPS','CLASSES','SOURCE_CLASSES_ATOMS').  Using these parameters, generate_facts dynamically produces queries and loads the table mom_candidate_facts.
filter_facts 
    PROCEDURE filter_facts(
        type_1              IN VARCHAR2,
        type_2              IN VARCHAR2,
        arg_1               IN VARCHAR2,
        arg_2               IN VARCHAR2,
        not_1               IN VARCHAR2,
        not_2               IN VARCHAR2,
        merge_set           IN VARCHAR2,
        authority           IN VARCHAR2,
        source              IN VARCHAR2,
        work_id             IN INTEGER
    );
This procedure is largely unimplemented but will eventually allow for a wide variety of filtering predicates.  It operates on the mom_candidate_facts removing those that meet the filtering criteria.  For a list of allowable filtering types and descriptions, see the code_map table where type='filter_type'.

replacement_merges 
    PROCEDURE replacement_merges(
        merge_set           IN VARCHAR2,
        normalization_flag  IN VARCHAR2,
        ambig_flag          IN VARCHAR2,
        authority           IN VARCHAR2,
        work_id             IN INTEGER
    );
This procedure implements a merging algorithm which sets concept_ids in source_classes_atoms to correspond with concept_ids in classes, and unsets the source_concept_status switches for concepts that no longer need to be inserted.  The normalization_flag allows for merging of higher concept_ids into lower ones or vice versa.  Use MEME_CONSTANTS.HIGHER_INTO_LOWER, or MEME_CONSTANTS.LOWER_INTO_HIGHER.  The ambig_flag allows you to require that merges take place only if the facts are 1-1, so 2 new concepts cannot merge into the same existing one.  Use a value of 0 if merging two concepts into the same concept is OK, and use 1 otherwise.

    PROCEDURE prepare_src_mergefacts(

        work_id             IN INTEGER

This procedure prepares source merge facts.  This procedure maps any id type found in code_map where table_name='sg_type'  in the mom_precomputed_fact tables into atom_ids and concept_ids.and then loads
data into mom_candidate_facts.
setup_partitions 
    PROCEDURE setup_partitions;
This procedure is unimplemented, but when we are using Oracle 8i, it should be called at the beginning of the source insertion process to create partitions in the core tables big enough to hold all of the new rows.  That way, editing can proceed while source insertion is taking place because the new and existing rows will be in different partitions.  One requirement might be that when replacing a source, all atoms from that source must be put into unaccessible partitions, otherwise pre-insert merge sets might have unstable concept ids.

update_releasability 
    PROCEDURE update_releasability(
        old_source          IN VARCHAR2,
        new_source          IN VARCHAR2,
        authority           IN VARCHAR2,
        new_value           IN VARCHAR2,
        work_id             IN INTEGER
    );
This procedure sets the tobereleased field in classes, relationships, and attributes for all rows connected to the source indicated in the old_source parameter. Typically, it is used to make an obsolete source unreleasable.  This procedure calls MEME_BATCH_ACTIONS.
resolve_stys 
    PROCEDURE resolve_stys(
        source              IN VARCHAR2,
        sty_fate            IN VARCHAR2,
        authority           IN VARCHAR2,
        work_id             IN INTEGER
    );
This procedure resolves records whose semantic types (STY's) are under consideration according to its fate, Win or Lose.  Fate W, means STY's from source win and others get deleted.  Fate L, means STY's from source
lose and get deleted.  This procedure calls MEME_BATCH_ACTIONS.
delete_demotions 
    PROCEDURE delete_demotions(

        work_id             IN INTEGER
    );
This procedure deletes demotions that exist between concepts that already have an approved concept level relationship if one of the atoms connected to the demotion is a safe replacement atom in the specified source.

map_obsolete_rels 
    PROCEDURE map_obsolete_rels( 
        stripped_source     IN VARCHAR2,
        authority           IN VARCHAR2,
        work_id             IN INTEGER
    );
This procedure replaces atom_ids of a relationship belonging to the obsolete version of the source specified in stripped_source with atom_ids belonging to the updated version of that same source. The process looks up all of the relationships connected to atoms of an old version of a source and finds which atoms in the new version of the source best replace old atoms.  This procedure calls MEME_BATCH_ACTIONS.
map_sg_fields 
    PROCEDURE map_sg_fields( 
        table_name          IN VARCHAR2,
        pair_flag           IN VARCHAR2 := MEME_CONSTANTS.NO,
        concept_flag        IN INTEGER := MEME_CONSTANTS.YES
    );
By default, this procedure requires a table that has atom_id, concept_id, sg_id, sg_type, and sg_qualifier fields (like source_attributes).  It maps the sg_id, given the type and qualifier to an atom_id and a concept_id.  If the concept_flag is MEME_CONSTANTS.NO, then the concept_id field is unnecessary and goes unmapped.  If the pair_flag is MEME_CONSTANTS.YES, then the table must have two sets of fields with the names above but with suffixes _1 and _2 (like atom_id_1, atom_id_2).
map_sg_data 
    PROCEDURE map_sg_data( 
        authority           IN VARCHAR2,
        work_id             IN VARCHAR2,
    );
This procedure re-maps all relationships and attributes that are natively connected to id types other than atom_ids or concept_ids.  All such rows are identified in the sg_relationships and sg_attributes tables..
